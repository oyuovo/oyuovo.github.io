引言 ​

缓存是后端性能优化的第一道防线，但引入缓存同时也引入了新的问题类别。缓存穿透、缓存雪崩、缓存击穿——这三个术语经常被提起，但很多文章只讲了概念，没有落到代码上。本文在解释每种问题原理的同时，给出可以直接运行的 Spring Boot 实现。

建议先阅读本文的概念部分，再结合代码理解——这些方案的生产化修改（如超时时间、锁粒度、布隆过滤器的误判率）高度依赖业务场景。

一、缓存穿透 ​

1.1 问题描述 ​

缓存穿透：请求查询的数据在缓存和数据库中都不存在。由于缓存不会命中，每次请求都穿透到数据库，造成数据库压力。

攻击者可以利用这一点，构造大量不存在的 ID 发起请求，轻易绕过缓存层。

1.2 方案一：缓存空值 ​

原理简单：数据库查询返回空时，将一个标记值存入缓存并设置较短的过期时间。

java
@Service
public class ProductService {

    @Autowired
    private StringRedisTemplate redisTemplate;
    @Autowired
    private ProductMapper productMapper;

    private static final String CACHE_PREFIX = "product:";
    private static final String NULL_PLACEHOLDER = "__NULL__";
    private static final Duration NULL_TTL = Duration.ofMinutes(2);    // 空值短 TTL
    private static final Duration DATA_TTL = Duration.ofHours(1);      // 正常数据长 TTL

    public Product getProductById(Long id) {
        String key = CACHE_PREFIX + id;
        String cached = redisTemplate.opsForValue().get(key);

        if (cached != null) {
            if (NULL_PLACEHOLDER.equals(cached)) {
                return null;  // 命中空值标记，直接返回
            }
            return JSON.parseObject(cached, Product.class);
        }

        // 缓存未命中，查数据库
        Product product = productMapper.selectById(id);

        if (product != null) {
            redisTemplate.opsForValue().set(key, JSON.toJSONString(product), DATA_TTL);
        } else {
            // 数据库也没有，缓存空值防止穿透
            redisTemplate.opsForValue().set(key, NULL_PLACEHOLDER, NULL_TTL);
        }
        return product;
    }
}

优缺点：

  • ✅ 实现简洁，与现有缓存层无缝集成
  • ❌ 大量不存在的 key 仍会占用 Redis 内存（通过短 TTL 缓解）
  • ❌ 如果攻击者不断构造新的不存在的 ID，缓存空值只能减缓而无法根除

1.3 方案二：布隆过滤器（推荐，彻底方案） ​

布隆过滤器是一个概率型数据结构，用于判断一个元素是否可能存在于集合中：

  • 判断为 「不存在」 → 一定不存在（100% 确定）
  • 判断为 「存在」 → 可能存在，也可能不存在（有误判率）

数学原理（简化版） ​

布隆过滤器维护一个长度为 m 的比特数组，初始全部为 0。添加元素时，用 k 个独立的哈希函数分别计算哈希值，将对应的 k 个比特位设为 1。查询时，计算 k 个哈希值，检查对应的 k 个比特位：

  • 如果任何一个 为 0 → 元素一定不存在
  • 如果全部 为 1 → 元素可能存在（有可能是其他元素的哈希碰撞导致的）

误判率 p 由以下公式近似：

p ≈ (1 - e^(-k*n/m))^k

其中 n = 预期元素数量，m = 比特数组长度，k = 哈希函数数量

Redisson 封装了布隆过滤器，使用一行 API 即可配置：

java
@Configuration
public class BloomFilterConfig {

    @Autowired
    private RedissonClient redissonClient;

    @Bean
    public RBloomFilter<Long> productBloomFilter() {
        RBloomFilter<Long> filter = redissonClient.getBloomFilter("bloom:product:id");
        // expectedInsertions: 预期元素数量（根据业务预估）
        // falseProbability: 可接受的误判率（业务可容忍的最大值）
        filter.tryInit(1_000_000L, 0.01);  // 100 万元素, 1% 误判率
        return filter;
    }
}

与缓存层集成 ​

java
@Service
public class ProductServiceWithBloom {

    @Autowired
    private StringRedisTemplate redisTemplate;
    @Autowired
    private ProductMapper productMapper;
    @Autowired
    private RBloomFilter<Long> bloomFilter;

    public Product getProductById(Long id) {
        // 1. 布隆过滤器快速判断：不存在 → 直接拒绝
        if (!bloomFilter.contains(id)) {
            return null;   // 100% 确定不存在，不查缓存也不查数据库
        }

        // 2. 可能存在 → 走正常缓存流程
        String key = "product:" + id;
        String cached = redisTemplate.opsForValue().get(key);
        if (cached != null) {
            return JSON.parseObject(cached, Product.class);
        }

        // 3. 查数据库并回写缓存
        Product product = productMapper.selectById(id);
        if (product != null) {
            redisTemplate.opsForValue().set(key, JSON.toJSONString(product),
                Duration.ofHours(1));
        }
        // 注意：如果 product == null，说明布隆过滤器误判了，概率 = 1%
        return product;
    }

    // 新增商品时同步添加到布隆过滤器
    public void addProduct(Product product) {
        productMapper.insert(product);
        bloomFilter.add(product.getId());  // 新 ID 加入过滤器
    }
}

布隆过滤器的局限性：

  • 不支持删除：标准布隆过滤器不能删除元素（将比特位从 1 翻回 0 会影响其他元素）。如果需要删除，考虑使用 Counting Bloom Filter。
  • 需要预判容量：tryInit 的 expectedInsertions 参数影响误判率——实际存储的元素数量超过预期时误判率会上升。建议按未来 2-3 年的增长预估。
  • Key 过期后的状态：如果数据库中的记录被删除，布隆过滤器中的对应比特位仍然为 1。对已删除 Key 的查询仍然会穿透缓存和数据库——这也是为什么很多系统将缓存空值与布隆过滤器结合使用。

二、缓存雪崩 ​

2.1 问题描述 ​

缓存雪崩：大量缓存在同一时间段内集中过期，或者 Redis 服务宕机，导致所有请求瞬间打到数据库。

造成雪崩的典型场景：

  1. 批量写入缓存时设置了相同的 TTL（如凌晨批量导入数据）
  2. Redis 集群故障导致缓存不可用
  3. 缓存预热不充分，服务重启后缓存为空

2.2 解决方案 ​

TTL 加随机偏移（最直接的预防） ​

java
private static final int BASE_TTL_MINUTES = 60;
private static final int JITTER_MINUTES = 10;  // 随机偏移 ±10 分钟
private static final ThreadLocalRandom RANDOM = ThreadLocalRandom.current();

private Duration randomizedTtl() {
    int minutes = BASE_TTL_MINUTES + RANDOM.nextInt(-JITTER_MINUTES, JITTER_MINUTES + 1);
    return Duration.ofMinutes(minutes);
}

// 使用
redisTemplate.opsForValue().set(key, value, randomizedTtl());

这样可以确保 key 不会在同一秒内集中过期——10 分钟的抖动窗口使过期时间分散在 50~70 分钟之间的任何时刻。

多级缓存 + 降级策略 ​

请求 → Caffeine 本地缓存（L1）
       → 命中：直接返回
       → 未命中：Redis（L2）
                  → 命中：回写 L1 + 返回
                  → 未命中：数据库
                            → 返回 + 回写 L2 + 回写 L1
                            → 异常（Redis 宕机或 DB 超时）：返回降级数据

多级缓存和降级逻辑是防御雪崩的纵深策略。具体实现见本博客的缓存架构相关文章。

三、缓存击穿 ​

3.1 问题描述 ​

缓存击穿（热点 Key 问题）：某个被高并发访问的 Key 在缓存失效的瞬间，大量请求同时打到数据库。

与穿透的区别：穿透查的是不存在的数据；击穿查的是存在但缓存刚好过期的数据。

与雪崩的区别：雪崩是大量 Key 同时过期；击穿是一个热点 Key 过期。

3.2 方案对比 ​

方案核心思路一致性性能复杂度
互斥锁缓存未命中时，只有获得锁的线程能查数据库并重建缓存，其他线程等待强一致等待期间阻塞低
逻辑过期不设物理 TTL，Value 中携带逻辑过期时间；过期时返回旧值并异步重建最终一致无阻塞中

选择指南：对一致性要求高的场景（库存扣减、余额查询）→ 互斥锁；对性能要求高且能容忍短暂不一致的场景（商品信息展示、文章内容）→ 逻辑过期。

3.3 互斥锁实现 ​

java
@Service
public class HotKeyMutexService {

    @Autowired
    private StringRedisTemplate redisTemplate;
    @Autowired
    private RedissonClient redissonClient;
    @Autowired
    private ProductMapper productMapper;

    public Product getProductWithMutex(Long productId) {
        String cacheKey = "product:" + productId;
        String lockKey = "lock:product:" + productId;

        // 1. 先查缓存
        String cached = redisTemplate.opsForValue().get(cacheKey);
        if (cached != null) {
            return JSON.parseObject(cached, Product.class);
        }

        RLock lock = redissonClient.getLock(lockKey);
        try {
            // 2. 获取分布式锁，最多等待 5 秒，锁持有时间 30 秒
            if (lock.tryLock(5, 30, TimeUnit.SECONDS)) {
                // 3. Double Check — 可能其他线程已经重建了缓存
                cached = redisTemplate.opsForValue().get(cacheKey);
                if (cached != null) {
                    return JSON.parseObject(cached, Product.class);
                }

                // 4. 查数据库并回写缓存
                Product product = productMapper.selectById(productId);
                if (product != null) {
                    redisTemplate.opsForValue().set(cacheKey,
                        JSON.toJSONString(product),
                        Duration.ofMinutes(60));
                }
                return product;
            } else {
                // 5. 获取锁超时 → 休眠重试或返回降级数据
                Thread.sleep(200);
                return getProductWithMutex(productId);
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            return null;
        } finally {
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}

互斥锁的关键细节：

  • Double Check：获得锁后必须再次检查缓存——在你排队等锁期间，前面的线程可能已经重建了缓存
  • 锁超时：tryLock 的等待超时防止请求堆积，锁持有超时防止死锁
  • isHeldByCurrentThread 判断：Redisson 的 WatchDog 机制会自动续期锁，但如果获得锁后线程因异常中断，锁可能已被释放。释放前确认锁仍属于当前线程

3.4 逻辑过期实现 ​

java
// 包装类：在 Value 中携带逻辑过期时间
@Data
@AllArgsConstructor
@NoArgsConstructor
public class RedisData<T> {
    private T data;
    private LocalDateTime expireTime;  // 逻辑过期时间
}
java
@Service
public class HotKeyLogicalExpireService {

    @Autowired
    private StringRedisTemplate redisTemplate;
    @Autowired
    private RedissonClient redissonClient;
    @Autowired
    private ProductMapper productMapper;

    private static final ExecutorService REBUILD_EXECUTOR =
        Executors.newFixedThreadPool(10);

    public Product getProductLogicalExpire(Long productId) {
        String cacheKey = "product:" + productId;
        String lockKey = "lock:product:" + productId;

        String cached = redisTemplate.opsForValue().get(cacheKey);
        if (cached == null) {
            return null;  // 缓存中没有这个 Key（不是热 Key，或者还没预热）
        }

        RedisData<Product> redisData = JSON.parseObject(cached,
            new TypeReference<RedisData<Product>>() {});

        // 判断是否逻辑过期
        if (LocalDateTime.now().isBefore(redisData.getExpireTime())) {
            return redisData.getData();   // 未过期，正常返回
        }

        // 逻辑过期 → 获取互斥锁
        RLock lock = redissonClient.getLock(lockKey);
        if (lock.tryLock()) {
            try {
                // Double Check（可能其他线程已经完成重建）
                cached = redisTemplate.opsForValue().get(cacheKey);
                RedisData<Product> doubleCheck = JSON.parseObject(cached,
                    new TypeReference<RedisData<Product>>() {});
                if (LocalDateTime.now().isBefore(doubleCheck.getExpireTime())) {
                    return doubleCheck.getData();
                }

                // 异步重建缓存
                REBUILD_EXECUTOR.submit(() -> {
                    try {
                        Product product = productMapper.selectById(productId);
                        RedisData<Product> newData = new RedisData<>(
                            product, LocalDateTime.now().plusHours(1));
                        redisTemplate.opsForValue().set(cacheKey,
                            JSON.toJSONString(newData));  // 注意：不设物理 TTL
                    } finally {
                        lock.unlock();
                    }
                });
            } catch (Exception e) {
                lock.unlock();
            }
        }
        // 无论是否获得锁，直接返回旧数据（最终一致性）
        return redisData.getData();
    }
}

逻辑过期的设计要点：

  • 不设物理 TTL：Redis key 永不过期（或设一个很长的物理 TTL 作为兜底），过期判断完全由业务代码中的 expireTime 字段控制
  • 异步重建：缓存重建在独立线程池中执行，请求线程直接返回旧数据——用户请求的响应时间不受缓存重建耗时的影响
  • 旧数据返回：在重建完成前，所有请求都拿到的是旧版本数据——这是「最终一致性」的代价

四、总结与决策框架 ​

请求到达
    │
    ├─→ 布隆过滤器判断 Key 是否存在？
    │       └─→ 不存在 → 直接返回 null（防穿透）
    │
    ├─→ 缓存命中？
    │       ├─→ 命中 + 未过期 → 返回
    │       ├─→ 命中 + 逻辑过期 → 返回旧值 + 异步重建（防击穿-逻辑过期方案）
    │       └─→ 未命中 → 获取互斥锁 → Double Check → 查 DB → 回写（防击穿-互斥锁方案）
    │
    └─→ TTL 设置：基础值 + 随机偏移（防雪崩）

三个问题不互斥——生产环境的缓存层通常会同时部署布隆过滤器（防穿透）+ 热点 Key 互斥锁（防击穿）+ TTL 随机化（防雪崩）。但引入的组件越多，运维复杂度越高。建议由简入繁：先上 TTL 随机化 → 再对已识别的热点 Key 加互斥锁 → 最后考虑布隆过滤器。
