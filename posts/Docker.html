引言 ​

Docker 对于后端开发者来说，不只是「把应用放进容器」——它改变了从本地开发到生产部署的整个工作流。本文假设你已经能运行 docker run 和 docker-compose up，聚焦于 Java/Spring Boot 开发者在 Docker 使用中真正需要关心的决策：多阶段构建如何减小镜像体积、Compose 如何编排本地开发依赖、JVM 在容器中的内存行为为何与裸机不同。

一、核心概念速览 ​

Docker 的三个核心抽象及对应类比：

Docker 概念类比说明
镜像（Image）类（Class）定义只读模板，包含运行应用所需的一切
容器（Container）实例（Instance）镜像的运行实例，有自己的文件系统层
仓库（Registry）Maven Central存储和分发镜像

与虚拟机对比的关键差异：Docker 容器共享宿主机内核，不需要 Guest OS。这带来两个结果——启动时间从分钟级降到秒级，以及 Linux 容器不能在 Windows/macOS 上原生运行（Docker Desktop 通过一个轻量 Linux VM 桥接）。

二、Spring Boot 应用的 Dockerfile 最佳实践 ​

2.1 初版 Dockerfile（不推荐） ​

dockerfile
FROM openjdk:17
COPY target/app.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]

问题：openjdk:17 是完整 JDK 镜像（~500MB），且包含了源码编译时需要的工具链（javac、jlink 等），运行时完全不需要。镜像膨胀的直接后果——每次 CI/CD 拉取镜像更慢、磁盘占用更大、攻击面更宽。

2.2 多阶段构建（推荐） ​

dockerfile
# ===== 阶段一：构建 =====
FROM maven:3.9-eclipse-temurin-17 AS builder
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline     # 先下载依赖（利用 Docker 缓存层）
COPY src/ ./src/
RUN mvn package -DskipTests

# ===== 阶段二：运行 =====
FROM eclipse-temurin:17-jre       # JRE 镜像，比 JDK 小 ~200MB
WORKDIR /app
COPY --from=builder /app/target/*.jar app.jar
EXPOSE 8080

# 非 root 用户（安全最佳实践）
RUN groupadd -r appuser && useradd -r -g appuser appuser
USER appuser

ENTRYPOINT ["java", "-jar", "app.jar"]

为什么这样写？

做法好处
多阶段构建构建工具（Maven）不进入最终镜像，镜像缩小 40-60%
先 COPY pom.xml 再 COPY src/Docker 的层缓存机制：源码变了但 pom.xml 没变 → 依赖层复用缓存
JRE 替代 JDK不需要编译器，JRE 就够了
非 root 用户容器逃逸后的最小权限原则，攻击者无法获取 root

2.3 分层打包（Spring Boot 2.3+） ​

Spring Boot 2.3 开始支持分层 JAR——将应用按变更频率拆分为多个层：

dockerfile
FROM eclipse-temurin:17-jre
WORKDIR /app
COPY target/*.jar app.jar
RUN java -Djarmode=layertools -jar app.jar extract  # 解压为分层目录

COPY --from=builder /app/dependencies/ ./
COPY --from=builder /app/spring-boot-loader/ ./
COPY --from=builder /app/snapshot-dependencies/ ./
COPY --from=builder /app/application/ ./

ENTRYPOINT ["java", "org.springframework.boot.loader.launch.JarLauncher"]
依赖层（变更最少 → 最常被缓存）
  ↓
Spring Boot loader（框架层，很少变更）
  ↓
快照依赖层
  ↓
应用代码层（每次构建都变更）

每次重新构建时，只有应用代码层需要重新 COPY，其他层全部命中 Docker 缓存——构建和推送速度提升 3-5 倍。

三、Docker Compose：本地开发栈 ​

3.1 为什么用 Compose？ ​

单独运行 docker run mysql + docker run redis + 手动管理网络 → 3 个终端窗口 + 每次重启都要记住参数。Compose 用一个 YAML 文件声明所有服务，docker compose up -d 一键启动整个本地环境。

3.2 Spring Boot 项目典型 Compose 配置 ​

yaml
# docker-compose.yml
services:
  mysql:
    image: mysql:8.0
    container_name: dev-mysql
    environment:
      MYSQL_ROOT_PASSWORD: root123
      MYSQL_DATABASE: myapp
    ports:
      - "3306:3306"
    volumes:
      - mysql-data:/var/lib/mysql              # 数据持久化
      - ./sql/init.sql:/docker-entrypoint-initdb.d/init.sql  # 自动初始化
    healthcheck:                               # 健康检查——应用等待 MySQL 就绪
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
      interval: 10s
      timeout: 5s
      retries: 5

  redis:
    image: redis:7-alpine
    container_name: dev-redis
    ports:
      - "6379:6379"
    volumes:
      - redis-data:/data
    command: redis-server --appendonly yes     # 开启 AOF 持久化

  app:
    build: .
    container_name: dev-app
    ports:
      - "8080:8080"
    environment:
      SPRING_DATASOURCE_URL: jdbc:mysql://mysql:3306/myapp
      SPRING_DATASOURCE_USERNAME: root
      SPRING_DATASOURCE_PASSWORD: root123
      SPRING_DATA_REDIS_HOST: redis
    depends_on:                                # 依赖声明（不等待健康检查）
      mysql:
        condition: service_healthy              # Compose 2.x: 等待健康检查通过
      redis:
        condition: service_started
    volumes:
      - ./target/app.jar:/app/app.jar           # 挂载本地构建产物（开发时直接替换 JAR）

volumes:
  mysql-data:
  redis-data:

三个关键设计：

  • depends_on + condition: service_healthy：确保 MySQL 完全就绪后才启动应用，避免连接失败
  • 容器间通过服务名通信：mysql:3306 而非 localhost:3306——Compose 自动创建网络并配置 DNS
  • 数据卷持久化：docker compose down 不会丢失数据库数据

四、容器网络：三种模式的选择 ​

模式行为使用场景
桥接（Bridge）容器拥有独立 IP，通过宿主机端口映射访问默认，99% 的场景用这个
Host容器直接使用宿主机网络栈需要最高网络性能且端口冲突可控
None容器没有网络安全隔离场景
bash
# 自定义桥接网络——同一网络内的容器可通过容器名互相访问
docker network create myapp-net
docker run -d --name mysql --network myapp-net mysql:8.0
docker run -d --name app --network myapp-net myapp:latest
# app 内部可通过 mysql:3306 连接数据库（容器名 = DNS 名）

五、JVM 容器感知：别让 OOM Killer 找上门 ​

5.1 JVM 在容器中的内存盲区 ​

Java 8 到 Java 10 早期版本中，JVM 不知道自己运行在容器内——它读取的是宿主机的内存信息，而不是容器的内存限制：

bash
# 容器限制 512MB，但 JVM 看到的是宿主机的 32GB
docker run -m 512m openjdk:8 java -XshowSettings:vm -version
# 输出: Max. Heap Size (Estimated): 7.8G  ← 远大于 512MB！
# 结果：GC 不积极 → 内存使用超出 cgroup limit → OOM Killer → 容器被 kill

5.2 解决方案 ​

Java 11+（推荐）：使用容器感知参数

bash
java -XX:+UseContainerSupport -XX:MaxRAMPercentage=75.0 -jar app.jar
# UseContainerSupport: 读取 cgroup 的内存限制而非宿主机内存
# MaxRAMPercentage=75%: 堆内存 = 容器内存的 75%（剩余 25% 留给 Metaspace、线程栈、堆外内存）

Dockerfile 中：

dockerfile
ENTRYPOINT ["java", \
    "-XX:+UseContainerSupport", \
    "-XX:MaxRAMPercentage=75.0", \
    "-XX:+ExitOnOutOfMemoryError", \    # OOM 时直接退出（让编排工具重启）
    "-jar", "app.jar"]

为什么不是 100%？

JVM 的内存使用 = 堆（Heap）+ Metaspace + 线程栈（Thread Stack）+ 堆外内存（Direct Buffer、Code Cache、GC 内部结构）。MaxRAMPercentage=75% 将堆限制在容器内存的 75%，剩下 25% 给非堆内存。如果设为 90%，当 Metaspace 或线程栈增长时就会触发容器 OOM。

Docker Compose 中设置资源限制：

yaml
services:
  app:
    image: myapp:latest
    deploy:
      resources:
        limits:
          memory: 512m           # 硬限制，超过触发 OOM Kill
        reservations:
          memory: 256m           # 软保证，调度时预留

七、总结 ​

决策点推荐做法一句话
基础镜像eclipse-temurin:17-jre 或 Alpine 版本JRE 够用不用 JDK，Alpine 更小但需注意 glibc 兼容
构建方式多阶段构建（Maven→JRE）镜像体积减半，CI/CD 加速
分层策略Spring Boot layertools + Docker 分层 COPY依赖层缓存 → 增量构建极快
ComposeMySQL + Redis + App 一套服务编排一键启动整个开发环境
JVM 参数UseContainerSupport + MaxRAMPercentage=75防止容器 OOM Killer
非 root 用户USER appuser最小权限原则

Docker 简化了环境一致性，但 JVM 与容器边界的交互需要主动关注——UseContainerSupport 和 MaxRAMPercentage 两个参数的配置，是 Java 应用容器化时最容易忽略也最致命的细节。
