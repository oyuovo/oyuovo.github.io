引言 ​

「学完 23 种设计模式，还是不知道代码怎么写。」

这不是你的问题。经典 GoF 设计模式诞生于 1994 年，C++ 时代。近 30 年后，Java 生态中的框架（特别是 Spring）已经内化了其中许多模式——你每天都在用它们，只是可能没意识到。

本文聚焦于 Java 后端开发中真正每天被使用的 7 种模式。对每种模式，回答三个问题：

  1. 它解决了什么问题？（而不是抽象定义）
  2. JDK / Spring 中哪里用了它？（让你看到模式的真实形态）
  3. 我在什么场景下应该主动用它？（可操作的指导）

一、单例模式（Singleton） ​

解决的问题 ​

确保一个类在整个应用中只有一个实例，并提供全局访问点。典型的场景：配置管理器、连接池、线程池——这些对象创建成本高，且多实例会导致状态不一致或资源浪费。

JDK / Spring 中的真实形态 ​

Spring 容器中的 Bean 默认就是单例的（@Scope("singleton")），但它与 GoF 单例模式有一个关键区别：

java
// GoF 单例：一个 JVM 中只有一个实例
public class ClassicSingleton {
    private static final ClassicSingleton INSTANCE = new ClassicSingleton();
    private ClassicSingleton() {}
    public static ClassicSingleton getInstance() { return INSTANCE; }
}

// Spring 单例：一个 ApplicationContext 中只有一个实例
// 两个 ApplicationContext 可以各有各的实例（GoF 单例做不到这一点）

Spring 的做法更灵活：单例的作用域是容器级别而非 JVM 级别。这在测试中尤为重要——每个测试用例可以启动独立的 ApplicationContext，互不干扰。

在 Java 中实现单例的最佳方式 ​

java
// 推荐方式：枚举单例（Josh Bloch 在 Effective Java 中推荐）
public enum ConfigManager {
    INSTANCE;

    private final Properties props = new Properties();

    ConfigManager() {
        // 加载配置文件
    }

    public String get(String key) {
        return props.getProperty(key);
    }
}

// 使用
String dbUrl = ConfigManager.INSTANCE.get("db.url");

枚举单例的三个优势：

  • 线程安全：JVM 保证枚举实例的初始化是线程安全的
  • 防反射攻击：Constructor.newInstance() 对枚举抛出 IllegalArgumentException
  • 防序列化破坏：反序列化自动返回同一个实例，不需要 readResolve()
什么时候不该用单例？

单例本质上是全局可变状态。如果单例是有状态的（维护计数器、缓存等），会导致：

  • 单元测试之间互相影响（测试 A 修改了单例状态，测试 B 的预期失效）
  • 隐式依赖（任何代码都可以访问单例，依赖关系无法从方法签名中看出）

如果 Spring Bean 已足够，不需要自己实现单例模式——让容器管理生命周期即可。

二、工厂模式（Factory） ​

解决的问题 ​

将对象的创建逻辑从调用方代码中分离。当对象的创建过程复杂（需要条件判断、配置组装、依赖注入）时，直接 new 会导致创建逻辑散落各处。

Spring 中的工厂模式 ​

Spring 容器本身就是一个巨大的工厂。BeanFactory 和 ApplicationContext 是工厂模式的经典应用：

java
// 不需要知道 UserService 的实现类或创建过程
UserService userService = applicationContext.getBean(UserService.class);

更精细的控制通过 FactoryBean 实现——当一个 Bean 的创建不能由简单的构造器完成时：

java
@Component
public class RestClientFactoryBean implements FactoryBean<RestClient> {

    @Value("${api.base-url}")
    private String baseUrl;

    @Override
    public RestClient getObject() {
        return RestClient.builder()
                .baseUrl(baseUrl)
                .defaultHeader("Accept", "application/json")
                .connectTimeout(Duration.ofSeconds(5))
                .build();  // 构造逻辑集中在一处
    }

    @Override
    public Class<?> getObjectType() {
        return RestClient.class;
    }
}

在业务代码中的使用场景 ​

当创建对象需要根据不同条件选择不同实现时，工厂方法可以消除 if-else 分支：

java
@Service
public class PaymentServiceFactory {

    private final Map<PaymentMethod, PaymentService> serviceMap;

    // Spring 自动将所有 PaymentService 实现注入到 Map 中
    // Key = Bean 名称, Value = Bean 实例
    public PaymentServiceFactory(Map<String, PaymentService> services) {
        this.serviceMap = services.stream()
            .collect(Collectors.toMap(
                PaymentService::supportedMethod,  // 每个实现声明自己支持哪种方式
                Function.identity()
            ));
    }

    public PaymentService getService(PaymentMethod method) {
        PaymentService service = serviceMap.get(method);
        if (service == null) {
            throw new IllegalArgumentException("Unsupported payment method: " + method);
        }
        return service;
    }
}

这比 if (method == WECHAT) return new WechatPayService() 更开放——新增支付方式只需添加一个实现类，不需要修改工厂代码。

三、策略模式（Strategy） ​

解决的问题 ​

将一组可互换的算法/行为封装为独立的类，让客户端可以在运行时选择。消除冗长的 if-else / switch 分支。

Spring 中的实现方式 ​

Spring 的依赖注入让策略模式极其简洁——利用 List 或 Map 自动收集接口的所有实现：

java
// 场景：根据用户等级计算不同的折扣
public interface DiscountStrategy {
    BigDecimal calculate(BigDecimal originalPrice);
    UserLevel supportedLevel();
}

@Component
public class VipDiscount implements DiscountStrategy {
    @Override
    public BigDecimal calculate(BigDecimal price) {
        return price.multiply(BigDecimal.valueOf(0.8));  // VIP 8 折
    }
    @Override
    public UserLevel supportedLevel() { return UserLevel.VIP; }
}

@Component
public class SuperVipDiscount implements DiscountStrategy {
    @Override
    public BigDecimal calculate(BigDecimal price) {
        return price.multiply(BigDecimal.valueOf(0.7));  // 超级 VIP 7 折
    }
    @Override
    public UserLevel supportedLevel() { return UserLevel.SUPER_VIP; }
}

@Service
public class PricingService {

    private final Map<UserLevel, DiscountStrategy> strategies;

    public PricingService(List<DiscountStrategy> strategyList) {
        this.strategies = strategyList.stream()
            .collect(Collectors.toMap(DiscountStrategy::supportedLevel, Function.identity()));
    }

    public BigDecimal getPrice(BigDecimal original, User user) {
        DiscountStrategy strategy = strategies.getOrDefault(
            user.getLevel(), price -> price);  // 默认无折扣
        return strategy.calculate(original);
    }
}

新增用户等级 → 新增一个 @Component 类 → 无需修改 PricingService。这是开闭原则（对扩展开放，对修改关闭）最自然的实现。

什么时候用策略模式，什么时候用 if-else？ ​

分支条件≤3 个且不太可能增加 → if-else 完全够用。分支可能在未来扩展（如支付渠道、通知方式、导出格式）→ 策略模式。

过度的策略模式是另一种形式的过度工程——5 行 if-else 拆成 5 个类文件并不会让代码更清晰。

四、模板方法模式（Template Method） ​

解决的问题 ​

定义算法的骨架，将某些步骤延迟到子类实现。子类可以在不改变算法结构的前提下重新定义某些步骤。

Spring 中的模板方法 ​

Spring 中大量使用模板方法模式——以 JdbcTemplate 为例：

java
// 你只需要提供变化的 SQL 和参数，连接管理、异常处理、资源释放由模板负责
@Autowired
private JdbcTemplate jdbcTemplate;

public User findById(Long id) {
    return jdbcTemplate.queryForObject(
        "SELECT * FROM users WHERE id = ?",
        (rs, rowNum) -> {   // 只有「如何将 ResultSet 映射为对象」是你需要写的
            User user = new User();
            user.setId(rs.getLong("id"));
            user.setUsername(rs.getString("username"));
            return user;
        },
        id
    );
}

Spring 生态中的模板方法应用：

框架模板类固定部分可变部分
Spring JDBCJdbcTemplate连接获取/释放、异常翻译SQL 语句 + 结果映射
Spring Data RedisRedisTemplate连接管理、序列化Redis 命令
Spring RESTRestTemplateHTTP 连接、请求/响应处理URL + 请求参数 + 响应体类型
Spring TransactionTransactionTemplate事务开启/提交/回滚业务操作

在业务代码中的使用 ​

java
// 场景：统一处理外部 API 调用（重试、日志、异常处理）
public abstract class ExternalApiTemplate<R, P> {

    private static final int MAX_RETRIES = 3;

    public R execute(P params) {
        for (int attempt = 1; attempt <= MAX_RETRIES; attempt++) {
            try {
                log.info("API 调用, 第{}次尝试", attempt);
                R result = doExecute(params);
                postProcess(result);
                return result;
            } catch (RetryableException e) {
                if (attempt == MAX_RETRIES) throw e;
                log.warn("API 调用失败, {}ms后重试", 1000L * attempt, e);
                sleep(1000L * attempt);
            }
        }
        throw new IllegalStateException("unreachable");
    }

    protected abstract R doExecute(P params);        // 子类实现：具体的 API 调用
    protected void postProcess(R result) {}           // 可选钩子：后处理（如保存日志）

    private void sleep(long ms) {
        try { Thread.sleep(ms); }
        catch (InterruptedException e) { Thread.currentThread().interrupt(); }
    }
}

模板方法 vs 策略模式：模板方法定义的是流程骨架（先做 A，再做 B，最后做 C，但步骤的实现可变）；策略模式定义的是可替换的完整算法（可以整体替换为另一套算法）。

五、责任链模式（Chain of Responsibility） ​

解决的问题 ​

将请求的发送者和接收者解耦，让多个处理者都有机会处理请求。每个处理者决定是自己处理还是传递给下一个。

Servlet Filter 链：最具代表性的 Java 实现 ​

java
// 每个 Filter 独立负责一个横切关注点
@Order(1)
@Component
public class RequestLogFilter implements Filter {
    @Override
    public void doFilter(ServletRequest req, ServletResponse resp, FilterChain chain) {
        log.info("请求进入: {}", ((HttpServletRequest) req).getRequestURI());
        chain.doFilter(req, resp);  // 交给下一个 Filter
        log.info("请求完成, 状态码: {}", ((HttpServletResponse) resp).getStatus());
    }
}

@Order(2)
@Component
public class AuthenticationFilter implements Filter {
    @Override
    public void doFilter(ServletRequest req, ServletResponse resp, FilterChain chain) {
        String token = ((HttpServletRequest) req).getHeader("Authorization");
        if (isValid(token)) {
            chain.doFilter(req, resp);            // 通过 → 继续
        } else {
            ((HttpServletResponse) resp).setStatus(401);  // 拦截 → 不继续
        }
    }
}

Spring Interceptor 使用相同的模式，只是位于 MVC 层而非 Servlet 层。

在业务代码中的使用 ​

责任链非常适合多步骤的校验/处理流程：

java
// 场景：订单提交前的多步校验
public interface OrderValidator {
    void validate(Order order) throws ValidationException;
}

@Component @Order(1)
public class StockValidator implements OrderValidator {
    public void validate(Order order) {
        if (order.getQuantity() > stockService.getAvailable(order.getProductId())) {
            throw new ValidationException("库存不足");
        }
    }
}

@Component @Order(2)
public class RiskValidator implements OrderValidator {
    public void validate(Order order) {
        if (order.getAmount().compareTo(new BigDecimal("100000")) > 0) {
            throw new ValidationException("大额订单需要人工审核");
        }
    }
}

@Service
public class OrderChainValidator {
    private final List<OrderValidator> validators;

    public OrderChainValidator(List<OrderValidator> validators) {
        this.validators = validators;  // Spring 自动收集所有实现
    }

    public void validate(Order order) {
        for (OrderValidator v : validators) {
            v.validate(order);  // 任一环节抛出异常则阻止提交
        }
    }
}

六、观察者模式（Observer / Pub-Sub） ​

解决的问题 ​

当一个对象的状态发生变化时，所有依赖它的对象自动收到通知并更新。这是实现 松耦合事件驱动 的核心模式。

Spring Event：开箱即用的观察者 ​

java
// 1. 定义事件
public class OrderCreatedEvent extends ApplicationEvent {
    private final Long orderId;
    public OrderCreatedEvent(Object source, Long orderId) {
        super(source);
        this.orderId = orderId;
    }
    public Long getOrderId() { return orderId; }
}

// 2. 发布事件
@Service
public class OrderService {
    @Autowired
    private ApplicationEventPublisher publisher;

    @Transactional
    public void createOrder(Order order) {
        orderMapper.insert(order);
        publisher.publishEvent(new OrderCreatedEvent(this, order.getId()));
        // 发布后继续执行，不等待监听者处理完成
    }
}

// 3. 监听事件
@Component
public class OrderNotificationListener {
    @EventListener
    public void handleOrderCreated(OrderCreatedEvent event) {
        // 发送通知、更新统计、同步搜索引擎...
        // 此方法在独立的事务中执行（如果加了 @Transactional + @Async）
    }
}

使用 @TransactionalEventListener 避免事务不一致

如果监听器在发布者事务提交前执行，可能出现「通知已发出但订单实际回滚了」的问题。使用 @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT) 确保监听器只在发布者事务成功提交后才执行。

什么时候升级到消息队列？ ​

Spring Event 是进程内的发布-订阅，事件和监听器在同一个 JVM 中。以下场景需要消息队列（RabbitMQ、Kafka 等）：

  • 事件的消费者不在同一个服务中（如订单服务 → 通知服务）
  • 需要持久化事件防止丢失（Broker 提供消息持久化）
  • 需要流量削峰（消费者可以按自己的速度消费）

简单法则：同一个微服务内部的事件协调 → Spring Event；跨服务的事件通信 → 消息队列。

七、装饰器模式（Decorator） ​

解决的问题 ​

在不修改原有类的情况下，动态地给对象添加额外的职责。这是对扩展开放，对修改关闭最经典的应用。

Java IO 中的装饰器 ​

java
// 每个装饰器添加一层功能，组合方式由调用方决定
InputStream rawInput = new FileInputStream("data.bin");
InputStream buffered = new BufferedInputStream(rawInput);       // 添加缓冲
InputStream uncompressed = new GZIPInputStream(buffered);       // 添加解压
ObjectInputStream objects = new ObjectInputStream(uncompressed);// 添加对象反序列化

Spring 中的装饰器：BeanPostProcessor ​

BeanPostProcessor 在 Bean 初始化前后进行包装，本质上是装饰器模式在容器级别的应用。AbstractAutoProxyCreator（AOP 代理的入口）就是通过 BeanPostProcessor 的 postProcessAfterInitialization 方法返回代理对象，为原始 Bean 装饰了事务、日志、权限等横切关注点。

java
// BeanPostProcessor 的本质：装饰原始 Bean
public interface BeanPostProcessor {
    default Object postProcessBeforeInitialization(Object bean, String beanName) {
        return bean;  // 默认原样返回
    }
    default Object postProcessAfterInitialization(Object bean, String beanName) {
        return bean;  // 可以返回装饰后的代理对象
    }
}

在业务代码中的使用 ​

java
// 场景：不修改原 Service 的情况下，为所有 Service 方法添加性能监控
// 使用 Spring AOP 实现（本质上是装饰器模式在代理层的应用）

@Aspect
@Component
public class MetricsAspect {

    @Around("execution(* com.example.service.*.*(..))")
    public Object recordMetrics(ProceedingJoinPoint joinPoint) throws Throwable {
        Timer.Sample sample = Timer.start(meterRegistry);
        try {
            return joinPoint.proceed();
        } finally {
            sample.stop(Timer.builder("service.calls")
                .tag("method", joinPoint.getSignature().toShortString())
                .register(meterRegistry));
        }
    }
}

八、总结：模式选择速查 ​

你遇到的问题推荐模式Spring / JDK 中的例证
对象创建复杂，或需要根据条件创建不同实现工厂模式BeanFactory, FactoryBean
需要全局唯一实例单例模式（枚举实现）Spring 的 singleton scope
多分支条件判断（if-else）且分支会扩展策略模式Spring 依赖注入 + List<Interface>
流程固定但某些步骤可变模板方法JdbcTemplate, RestTemplate
请求经过多层校验/处理，每层独立责任链Servlet Filter 链, Spring Interceptor
一个对象变化，多个对象需要响应观察者模式Spring Event, @EventListener
动态给对象添加功能而不修改原有代码装饰器模式Java IO 流, BeanPostProcessor, AOP

设计模式不是目的，而是识别代码中常见问题并应用已验证方案的工具。当你看到一段代码有 10 个 if-else 分支、或者每次加功能都要改核心类、或者多个类承担了相同的职责——那就是模式该介入的时刻。但同样重要的是：如果今天的问题用 5 行直白的代码就能解决，不要为了「用上模式」而引入 3 个接口和 5 个类。
