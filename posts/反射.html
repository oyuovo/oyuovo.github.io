引言 ​

反射是 Java 中最强大的运行时能力之一——它允许你在不知道类名的情况下创建实例、调用方法、访问字段。Spring 的依赖注入、MyBatis 的 Mapper 绑定、Jackson 的 JSON 序列化——都建立在反射之上。

但反射也有代价：性能开销、类型安全丧失、以及 Java 9+ 模块系统引入的访问限制。本文从 API 出发，深入到性能分析和现代替代方案。

一、反射 API 核心操作 ​

1.1 获取 Class 对象的三种方式 ​

java
// 方式一：类字面量（编译时已知类型）
Class<String> cls = String.class;

// 方式二：实例的 getClass()（已有对象时）
String s = "hello";
Class<? extends String> cls2 = s.getClass();

// 方式三：全限定名（运行时动态加载——最常用）
Class<?> cls3 = Class.forName("com.example.model.User");

Class.forName() 是框架中最常用的方式——用于加载在编译时未知的实现类（如从配置文件中读取类名）。

1.2 操作字段、方法和构造器 ​

java
// 获取 User 类的 Class 对象
Class<?> cls = Class.forName("com.example.model.User");

// ---- 字段 ----
Field nameField = cls.getDeclaredField("username");
nameField.setAccessible(true);                    // 私有字段也需要这个
String name = (String) nameField.get(userInstance);
nameField.set(userInstance, "newUsername");

// ---- 方法 ----
Method setNameMethod = cls.getDeclaredMethod("setUsername", String.class);
setNameMethod.invoke(userInstance, "newValue");   // 等价于 user.setUsername("newValue")

// ---- 构造器 ----
Constructor<?> constructor = cls.getDeclaredConstructor(Long.class, String.class);
Object newUser = constructor.newInstance(1L, "Alice");

setAccessible(true) 绕过 Java 的访问控制（private/protected），这是框架能够注入私有字段的原因。从 Java 9 开始，模块（module）未显式 opens 的包，setAccessible 会抛出 InaccessibleObjectException。

二、反射性能：不是一个数量级 ​

反射慢在哪里？主要开销来自三部分：

  1. 元数据查找：getDeclaredMethod("methodName", paramTypes...) 需要遍历类的方法表，按名称和参数类型匹配
  2. 访问检查：每次 invoke 都要检查访问权限（即使调用了 setAccessible）
  3. 参数装箱/拆箱：invoke(Object target, Object... args) 的参数和返回值都是 Object，每个原始类型参数都要装箱
java
// 性能基准（百万次调用）
// 直接调用:                ~5ms
// 反射（每次 getMethod）:  ~350ms  (70× 慢)
// 反射（缓存 Method）:     ~80ms   (16× 慢)
// MethodHandle:            ~20ms   (4× 慢)
// LambdaMetafactory:       ~6ms    (接近直接调用)
java
// MethodHandle —— Java 7+ 的反射替代
MethodHandles.Lookup lookup = MethodHandles.lookup();
MethodType mt = MethodType.methodType(void.class, String.class);
MethodHandle handle = lookup.findVirtual(User.class, "setUsername", mt);
handle.invokeExact(userInstance, "newName");  // 无装箱！签名必须严格匹配

// LambdaMetafactory —— 生成函数式接口实例，性能最接近直接调用
// 常用于高性能框架（如 Jackson 的序列化器）

选择指南：

  • 框架开发（Spring/MyBatis 级别）：反射 + 缓存 Method → 一次性查找开销可接受
  • 高频调用路径：MethodHandle → 无装箱，性能显著好于反射
  • 极致性能（序列化、ORM）：LambdaMetafactory → 生成的一等公民方法引用，JIT 完全优化

三、反射的现代替代方案 ​

3.1 当类型已知时——不要用反射 ​

如果你的代码已经知道具体类型，反射就是多余的：

java
// ❌ 不必要的反射
Class<?> cls = UserService.class;
Method method = cls.getMethod("getUserById", Long.class);
Object result = method.invoke(userService, userId);

// ✅ 直接调用
User user = userService.getUserById(userId);

反射的唯一合理使用场景是：类型在编译时不可知。

3.2 Java 17+ 模块系统的访问限制 ​

java
// 以下代码在模块化应用中可能失败
Field field = String.class.getDeclaredField("value");  // String 的内部字段
field.setAccessible(true);  // ❌ InaccessibleObjectException

解决方案（由 JVM 启动参数指定）：

bash
# 为特定模块开放反射访问
--add-opens java.base/java.lang=ALL-UNNAMED

这是 Java 平台安全模型的进步——核心类库的内部实现不再无条件对反射开放。迁移到 Java 17+ 的应用如果依赖了操作 JDK 内部字段的第三方库，需要关注启动参数配置。

四、框架中的反射：两个经典案例 ​

4.1 Spring 的依赖注入 ​

java
// Spring 内部简化版 Bean 创建流程
public <T> T createBean(Class<T> clazz) {
    // 1. 通过反射获取构造器
    Constructor<T> constructor = clazz.getDeclaredConstructor();
    T instance = constructor.newInstance();

    // 2. 遍历所有 @Autowired 字段，反射注入
    for (Field field : clazz.getDeclaredFields()) {
        if (field.isAnnotationPresent(Autowired.class)) {
            field.setAccessible(true);
            Object dependency = getBean(field.getType());
            field.set(instance, dependency);     // 反射赋值
        }
    }
    return instance;
}

4.2 MyBatis 的 Mapper 代理 ​

MyBatis 的 UserMapper 是一个接口——你从来没有写过它的实现类。MyBatis 使用 JDK 动态代理在运行时生成实现：

java
// 核心思路（极度简化）
UserMapper mapper = (UserMapper) Proxy.newProxyInstance(
    UserMapper.class.getClassLoader(),
    new Class<?>[] { UserMapper.class },
    (proxy, method, args) -> {
        // method = getUserById
        // args = [1L]
        String sql = parseAnnotation(method);    // 反射读 @Select 注解
        return jdbcTemplate.query(sql, args);    // 执行 SQL
    }
);

这两个案例都满足「编译时类型不可知」的条件——Spring 在配置时不知道用户定义的 Bean 类型，MyBatis 在编译时不知道 Mapper 接口的 SQL 映射。

五、总结 ​

维度结论一句话
使用场景类型在编译时不可知「我知道类型」→ 不需要反射
性能直接调用 > LambdaMetafactory > MethodHandle >> 反射框架 OK，热路径用 MethodHandle
缓存Method 实例务必缓存getMethod() 不要在循环里调用
模块系统Java 17+ 需 --add-opensJDK 内部 API 不再无条件可反射
现代替代方法引用、Lambda > 反射能用一等公民就不要用反射

反射是框架的基础设施，但不是业务代码的工具。如果你的业务代码中出现了 getDeclaredMethod().invoke()，大概率是你选择了一个错误的设计方案。
