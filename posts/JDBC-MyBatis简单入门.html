引言 ​

JDBC 是 Java 数据库访问的底层基石，但你不需要直接用它——就像你不需要直接操作 Socket 来发 HTTP 请求。理解一层抽象「为什么存在」比记忆它的 API 更重要。本文直接进入 MyBatis/MyBatis-Plus 的实战用法，聚焦于后端开发中每天遇到的问题：动态 SQL 怎么写不变成一团乱麻、关联查询的 N+1 怎么发现和解决、慢 SQL 怎么自动拦截。

一、为什么不直接写 JDBC？ ​

JDBC 的核心流程：

java
// 6 个步骤，20+ 行代码，只完成一条 SELECT
Connection conn = DriverManager.getConnection(url, user, password);
PreparedStatement stmt = conn.prepareStatement("SELECT * FROM users WHERE id = ?");
stmt.setLong(1, userId);
ResultSet rs = stmt.executeQuery();
while (rs.next()) {
    User user = new User();
    user.setId(rs.getLong("id"));
    user.setUsername(rs.getString("username"));    // 字段名是字符串，拼写错误运行时才知道
    // ...
}
rs.close();
stmt.close();
conn.close();   // 漏了？→ 连接泄漏

这些问题催生了 MyBatis：

  • SQL 与 Java 代码分离：SQL 写在 XML 或注解中，不被 Java 字符串拼接掩盖
  • 自动结果映射：ResultSet → Java 对象，消除手动 set
  • 动态 SQL：根据条件拼接 WHERE 子句，不手写 if-else 拼字符串
  • 连接管理：与连接池（HikariCP）集成，自动获取和释放连接

二、动态 SQL：MyBatis 最强大的特性 ​

2.1 where + if：条件查询 ​

xml
<select id="findUsers" resultType="User">
    SELECT id, username, email, status, created_at
    FROM users
    <where>
        <if test="keyword != null and keyword != ''">
            AND (username LIKE CONCAT('%', #{keyword}, '%')
                 OR email LIKE CONCAT('%', #{keyword}, '%'))
        </if>
        <if test="status != null">
            AND status = #{status}
        </if>
        <if test="roleIds != null and roleIds.size() > 0">
            AND role_id IN
            <foreach collection="roleIds" item="roleId" open="(" close=")" separator=",">
                #{roleId}
            </foreach>
        </if>
    </where>
    ORDER BY created_at DESC
</select>

<where> 标签自动处理第一个条件的 AND 前缀——如果没有匹配的条件，整个 WHERE 子句不输出。这个细节是动态 SQL 从可用到好用的关键。

2.2 choose-when：分支选择 ​

xml
<select id="findByCondition" resultType="User">
    SELECT * FROM users
    <where>
        <choose>
            <when test="id != null">
                id = #{id}                     <!-- 精确查询，最高优先级 -->
            </when>
            <when test="email != null">
                email = #{email}
            </when>
            <otherwise>
                status = 'ACTIVE'              <!-- 默认条件 -->
            </otherwise>
        </choose>
    </where>
</select>

等价于 Java 中的 if (id != null) { ... } else if (email != null) { ... } else { ... }。避免了大量 <if test="cond1">...<if test="!cond1 and cond2"> 的嵌套。

三、N+1 查询问题与解决方案 ​

3.1 什么是 N+1 查询？ ​

xml
<!-- 查询订单列表（1 次查询） -->
<select id="findOrders" resultMap="OrderResultMap">
    SELECT * FROM orders WHERE user_id = #{userId}
</select>

<!-- 对每个订单，查询订单项（N 次查询） -->
<select id="findItemsByOrderId" resultType="OrderItem">
    SELECT * FROM order_items WHERE order_id = #{orderId}
</select>

<resultMap id="OrderResultMap" type="Order">
    <collection property="items" column="id"
                select="findItemsByOrderId"/>   <!-- ← 每行触发一次子查询 -->
</resultMap>

结果：1 条查询拿到 50 个订单 → 50 条子查询拿到每个订单的 Item → 总共 51 条 SQL。

3.2 解决方案：嵌套结果映射 ​

xml
<resultMap id="OrderResultMap" type="Order">
    <id property="id" column="order_id"/>
    <result property="orderDate" column="order_date"/>
    <collection property="items" ofType="OrderItem"
                resultMap="OrderItemResultMap"/>   <!-- 一次 JOIN 全部搞定 -->
</resultMap>

<select id="findOrdersWithItems" resultMap="OrderResultMap">
    SELECT
        o.id AS order_id,
        o.order_date,
        oi.id AS item_id,
        oi.product_name,
        oi.quantity,
        oi.unit_price
    FROM orders o
    LEFT JOIN order_items oi ON o.id = oi.order_id
    WHERE o.user_id = #{userId}
</select>

一次 JOIN 取代了 N+1 条查询。MyBatis 根据 id 标签自动将 JOIN 结果中的重复订单行合并为带 List<OrderItem> 的 Order 对象。

何时用嵌套查询，何时用 JOIN？

  • 关联数据一定会被使用（如展示订单详情必须带上订单项）→ JOIN 嵌套结果
  • 关联数据有时才需要（如订单列表不展示订单项，只有点击详情才加载）→ 懒加载（fetchType=LAZY）+ 嵌套查询

关键决策点：如果一次页面渲染需要主对象和关联对象 → JOIN；如果关联对象是「可选」的 → 懒加载。

四、MyBatis-Plus：让你的 Mapper 自动 CRUD ​

java
// 接口继承 BaseMapper，免费获得 CRUD 方法
public interface UserMapper extends BaseMapper<User> {
    // 如果需要自定义查询，添加方法即可
    @Select("SELECT * FROM users WHERE email = #{email}")
    User selectByEmail(String email);
}

// 自带方法：
// userMapper.insert(user)
// userMapper.selectById(1L)
// userMapper.selectList(wrapper)  — wrapper 是条件构造器
// userMapper.updateById(user)
// userMapper.deleteById(1L)
java
// 条件构造器：类型安全的动态查询
List<User> users = userMapper.selectList(
    new LambdaQueryWrapper<User>()
        .like(User::getUsername, keyword)
        .eq(User::getStatus, "ACTIVE")
        .gt(User::getCreatedAt, LocalDate.now().minusDays(7))
        .orderByDesc(User::getCreatedAt)
);

LambdaQueryWrapper 使用 Lambda 表达式引用字段名，重构时字段名变化会自动更新——避免 XML 中字符串形式的字段名在重命名后静默失败。

五、慢 SQL 拦截器 ​

java
@Component
@Intercepts({
    @Signature(type = StatementHandler.class,
               method = "prepare", args = {Connection.class, Integer.class})
})
public class SlowSqlInterceptor implements Interceptor {

    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        long start = System.currentTimeMillis();
        try {
            return invocation.proceed();
        } finally {
            long elapsed = System.currentTimeMillis() - start;
            if (elapsed > 1000) {  // 超过 1 秒即为慢 SQL
                StatementHandler handler = (StatementHandler) invocation.getTarget();
                String sql = handler.getBoundSql().getSql()
                    .replaceAll("\\s+", " ");
                log.warn("慢SQL [{}ms]: {}", elapsed, sql);
            }
        }
    }
}

MyBatis-Plus 内置了性能分析插件（PerformanceInterceptor），但在 3.2.0 后已移除——推荐使用 Druid 的 Filter 或 P6Spy 做 SQL 监控，功能更完整。

六、总结 ​

实践推荐一句话
动态 SQLXML <where> + <if>不要用 Java 字符串拼接 SQL
N+1 查询优先嵌套结果（JOIN），按需懒加载用日志或 Druid 监控发现 N+1
基础 CRUDMyBatis-Plus BaseMapper不手写简单的增删改查
条件构造LambdaQueryWrapper字段名安全重构
慢 SQL拦截器 + Druid SQL 监控1 秒以上 SQL 必须告警

MyBatis 的精髓在于「SQL 的控制权在开发者手中」——你可以写出最优化的 SQL，同时借助框架消除 JDBC 的样板代码。真正用好 MyBatis 的标志是：写出的 SQL 和你在数据库客户端中调试出来的一样精确。
