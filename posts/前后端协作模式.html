引言 ​

后端开发者与前端协作时，有三个问题是绕不开的：前端怎么发请求、跨域怎么解决、接口没写好时怎么模拟数据。本文将这三个问题串成一条线，既讲前端 Axios 的工程化封装，也讲后端 CORS 的工作原理（不只是复制粘贴配置），最后讨论前后端 API 契约设计——如何让双方在对接口时少吵架。

一、Axios 工程化封装 ​

1.1 为什么需要封装？ ​

直接在每个组件里写 axios.get(url).then(...) 在两个组件之后就失控了：

  • 请求根路径写死在每个调用点，环境切换时改动遍地
  • Token 过期处理逻辑分散在各处，遗漏一处就是一个 Bug
  • 错误提示样式不统一

一个规范的 Axios 实例应该包含三层：实例创建 → 请求拦截 → 响应拦截。

1.2 生产级 Axios 实例 ​

ts
// api/request.ts
import axios, { AxiosError, InternalAxiosRequestConfig, AxiosResponse } from 'axios'
import { ElMessage } from 'element-plus'
import router from '@/router'

const http = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL || '/api',
  timeout: 15000,                                 // 15s 超时，比默认的永远等待好
  headers: { 'Content-Type': 'application/json' }
})

// ===== 请求拦截器 =====
http.interceptors.request.use(
  (config: InternalAxiosRequestConfig) => {
    const token = localStorage.getItem('access_token')
    if (token) {
      config.headers.Authorization = `Bearer ${token}`
    }
    return config
  },
  (error: AxiosError) => Promise.reject(error)
)

// ===== 响应拦截器 =====
http.interceptors.response.use(
  (response: AxiosResponse) => {
    // 直接返回 data，调用方不需要每次都 .data
    return response.data
  },
  async (error: AxiosError) => {
    const { response } = error

    if (response?.status === 401) {
      // Token 过期 → 尝试用 refresh_token 续期
      const refreshed = await tryRefreshToken()
      if (refreshed) {
        // 续期成功，重试原始请求
        return http(error.config!)
      }
      // 续期失败，踢回登录页
      localStorage.clear()
      router.push('/login')
      ElMessage.error('登录已过期')
    }

    // 统一错误提示
    const msg = (response?.data as any)?.message || '网络错误，请稍后重试'
    ElMessage.error(msg)

    return Promise.reject(error)
  }
)

export default http

拦截器的设计关键点：

拦截器职责不应做的事
请求拦截添加 Token、添加公共请求头、请求日志修改业务参数
响应拦截剥离 .data、统一错误提示、Token 刷新吞掉错误（调用方可能需要区分错误类型）

1.3 按业务模块组织 API ​

ts
// api/modules/user.ts
import http from '../request'
import type { PaginatedResult } from '@/types'

export interface UserVO {
  id: number
  username: string
  email: string
  roles: string[]
  createdAt: string
}

export const userApi = {
  list(params: { page: number; size: number; keyword?: string }) {
    return http.get<PaginatedResult<UserVO>>('/api/admin/users', { params })
  },
  getById(id: number) {
    return http.get<UserVO>(`/api/admin/users/${id}`)
  },
  create(data: Partial<UserVO>) {
    return http.post<UserVO>('/api/admin/users', data)
  },
  update(id: number, data: Partial<UserVO>) {
    return http.put<UserVO>(`/api/admin/users/${id}`, data)
  },
  delete(id: number) {
    return http.delete(`/api/admin/users/${id}`)
  }
}

每个业务域一个模块文件，导入路径清晰，IDE 自动补全所有可用接口。这与后端的 Controller → Service → Mapper 分层思路完全一致。

二、CORS 跨域：不只是复制粘贴配置 ​

2.1 同源策略：浏览器的安全基线 ​

浏览器强制执行的同源策略（Same-Origin Policy）规定：只有当协议、域名、端口三者完全相同时，脚本才能自由读写对方资源。这是浏览器安全的基石——如果没有同源策略，任何网站都能通过你的浏览器向你的银行发起请求。

跨域判断示例（假设当前页面在 http://localhost:5173）：

目标 URL是否跨域原因
http://localhost:5173/api/user❌ 不跨域同源
http://localhost:8080/api/user✅ 跨域端口不同
https://localhost:5173/api/user✅ 跨域协议不同
http://api.example.com/user✅ 跨域域名不同

2.2 CORS 的工作机制 ​

CORS（Cross-Origin Resource Sharing）是 W3C 标准，通过在 HTTP 头部增加字段，让服务器明确声明允许哪些源站访问。

简单请求 ​

同时满足以下条件才是「简单请求」：

  • 方法为 GET、POST、HEAD 之一
  • 仅包含三类安全请求头：Accept、Accept-Language、Content-Language、Content-Type
  • Content-Type 限定为 text/plain、multipart/form-data、application/x-www-form-urlencoded

简单请求的 CORS 流程：

浏览器                                        服务器
   │  GET /api/user HTTP/1.1                   │
   │  Origin: http://localhost:5173            │
   │─────────────────────────────────────────→ │
   │                                            │  检查 Origin 是否在白名单中
   │  HTTP/1.1 200 OK                           │
   │  Access-Control-Allow-Origin: localhost    │
   │←───────────────────────────────────────── │
   │                                            │
   │  浏览器对比 Origin 和 Allow-Origin          │
   │  匹配 → 放行；不匹配 → 拦截                 │

非简单请求与预检（Preflight） ​

当请求使用了 PUT/DELETE 方法，或 Content-Type: application/json（这正是前后端分离中最常见的场景），浏览器会在真实请求之前发送一个 OPTIONS 预检请求：

浏览器                                        服务器
   │  OPTIONS /api/user HTTP/1.1                │
   │  Origin: http://localhost:5173            │
   │  Access-Control-Request-Method: PUT       │
   │  Access-Control-Request-Headers:          │
   │    Content-Type, Authorization             │
   │─────────────────────────────────────────→ │
   │                                            │
   │  HTTP/1.1 204 No Content                   │
   │  Access-Control-Allow-Origin: *            │
   │  Access-Control-Allow-Methods: GET,PUT     │
   │  Access-Control-Allow-Headers:             │
   │    Content-Type, Authorization              │
   │  Access-Control-Max-Age: 86400            │ ← 24h 内同样的请求不再预检
   │←───────────────────────────────────────── │
   │                                            │
   │  预检通过 → 发送真实 PUT 请求              │

Access-Control-Max-Age 的实际意义

预检请求每次都要多一次 HTTP 往返，在高延迟网络中影响显著。设置合理的 max-age（如 3600 秒 = 1 小时）让浏览器缓存预检结果，减少不必要的 OPTIONS 请求。这个配置在生产环境尤为重要。

2.3 Spring Boot 中的 CORS 配置 ​

java
@Configuration
public class CorsConfig implements WebMvcConfigurer {

    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")            // 仅对 API 路径启用
                .allowedOriginPatterns("*")        // 生产环境应指定具体域名
                .allowedMethods("GET", "POST", "PUT", "DELETE", "OPTIONS")
                .allowedHeaders("*")
                .allowCredentials(true)            // 允许携带 Cookie
                .maxAge(3600);                     // 预检缓存 1 小时
    }
}

生产环境注意事项

  • allowedOriginPatterns("*") 与 allowCredentials(true) 不能同时使用（浏览器安全限制）——实际上 allowedOriginPatterns 支持通配符且兼容 credentials，而 allowedOrigins("*") 不兼容
  • 生产环境应将 * 替换为具体的前端域名列表
  • 如果使用了 Spring Security，还需在 SecurityFilterChain 中显式调用 .cors(),否则 CorsConfig 不会生效

三、前端 Mock 方案选型 ​

3.1 Mock.js 的困境 ​

Mock.js 长期以来是前端数据模拟的首选工具，但它有一个根本性的问题：通过重写 XMLHttpRequest 拦截请求。这意味着：

  • 请求根本没有发出，开发者工具的 Network 面板看不到任何请求记录
  • 无法测试真实的请求/响应流程
  • 与 Axios 的新版本偶尔存在兼容性问题
  • 项目已多年未实质性更新

3.2 MSW：现代 Mock 方案 ​

Mock Service Worker (MSW) 是当前社区推荐的数据模拟方案。它在 Service Worker 层拦截网络请求，与业务代码完全解耦：

业务代码 (Axios)
    │
    ▼
MSW Service Worker  ──→ 匹配 Mock Handler ──→ 返回模拟数据
    │                                            （请求未离开浏览器）
    ▼
真实网络请求（如果不匹配任何 Mock）

MSW 的核心优势：

特性Mock.jsMSW
拦截位置重写 XMLHttpRequestService Worker
Network 面板可见❌ 不可见✅ 可见（标记为 Service Worker）
与框架无关✅✅
REST API + GraphQL仅 REST两者都支持
测试环境复用不支持同一套 handler 可用于开发 + 测试
维护活跃度几乎停滞活跃维护，2024 年发布 2.x

3.3 MSW 基础使用 ​

bash
npm install msw --save-dev
npx msw init public/ --save   # 在 public 目录生成 Service Worker
ts
// mocks/handlers.ts
import { http, HttpResponse } from 'msw'

export const handlers = [
  // 拦截 GET /api/user/:id
  http.get('/api/user/:id', ({ params }) => {
    return HttpResponse.json({
      code: 200,
      data: {
        id: Number(params.id),
        username: 'oyuovo',
        email: 'oyuovo@example.com',
        roles: ['USER']
      }
    })
  }),

  // 拦截 POST /api/auth/login
  http.post('/api/auth/login', async ({ request }) => {
    const body = await request.json() as { username: string }
    return HttpResponse.json({
      code: 200,
      data: { token: 'mock-jwt-token-' + body.username }
    })
  })
]
ts
// mocks/browser.ts
import { setupWorker } from 'msw/browser'
import { handlers } from './handlers'

export const worker = setupWorker(...handlers)

// 仅在开发环境启动
if (import.meta.env.DEV) {
  worker.start({ onUnhandledRequest: 'bypass' })
    // onUnhandledRequest: 'bypass' 表示未匹配的请求正常发出
}

3.4 当后端未就绪时的工作流 ​

  1. 前后端对齐 API 接口文档（Swagger / Apifox）
  2. 前端根据文档编写 MSW handler
  3. 前端使用 MSW 模拟数据独立开发
  4. 后端接口就绪后，关闭 MSW，切换到真实接口
  5. 联调阶段只需微调不符合预期的字段名

这套流程的核心收益是：前后端并行开发，互不阻塞。MSW handler 本质上是接口文档的可执行版本——如果文档说返回 {userId: 1}，但实际返回了 {user_id: 1}，MSW 阶段就能发现字段名不一致。

四、API 契约设计：减少联调摩擦 ​

4.1 统一响应格式 ​

前后端约定一个统一的响应包结构，是所有协作的基础：

json
{
  "code": 200,
  "message": "success",
  "data": { ... }
}
java
// 后端统一响应体
@Data
public class ApiResponse<T> {
    private int code;
    private String message;
    private T data;

    public static <T> ApiResponse<T> ok(T data) {
        return new ApiResponse<>(200, "success", data);
    }

    public static <T> ApiResponse<T> fail(int code, String message) {
        return new ApiResponse<>(code, message, null);
    }
}
ts
// 前端统一响应类型
interface ApiResponse<T> {
  code: number
  message: string
  data: T
}

4.2 命名规范与分页约定 ​

事项建议反例
字段命名camelCase（JSON 惯例）user_id、UserName
日期格式ISO 8601 字符串 "2025-11-26T10:30:00Z"1713800000000（时间戳，人不可读）
分页参数page (从 1 开始)、size、返回 total/recordsoffset/limit 与 page/size 混用
空值处理null，不省略字段字段不存在 vs 字段为 null 语义混淆
ts
// 统一分页响应类型
interface PaginatedResult<T> {
  records: T[]
  total: number
  page: number
  size: number
}

4.3 错误码设计 ​

HTTP 状态码只分大类（200/400/401/403/404/500），具体的错误原因由业务码（code）承载：

java
public enum ErrorCode {
    USER_NOT_FOUND(10001, "用户不存在"),
    PASSWORD_ERROR(10002, "密码错误"),
    TOKEN_EXPIRED(10003, "Token 已过期"),
    PERMISSION_DENIED(10004, "权限不足"),
    DUPLICATE_ENTRY(20001, "数据重复"),
    STOCK_INSUFFICIENT(30001, "库存不足");
}

错误码按模块分段（10xxx 认证模块，20xxx 数据校验，30xxx 业务规则），便于快速定位问题归属。前端根据 code 做差异化处理（10003 → 刷新 Token，10004 → 跳转无权限页，其他 → toast 提示）。

五、总结 ​

前后端协作的质量取决于三个层次的约定：

层次内容关键决策
传输层Axios 封装、Token 管理、CORS用拦截器集中处理，不要分散在各组件
数据层Mock 方案、响应格式、命名规范新项目选 MSW 替代 Mock.js
语义层错误码体系、分页约定、状态同步错误码分段管理，按模块划分区间

两端遵循同一套约定，联调的节奏会从「这个字段怎么没有？」变成「这个接口有 1 个字段名需要对齐」——后者通常 5 分钟就能解决。
