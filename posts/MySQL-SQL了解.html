引言 ​

超过 90% 的 SQL 问题出现在后 10% 的 SQL 知识上。SELECT * FROM orders WHERE status = 'PAID' ORDER BY created_at LIMIT 10 这种查询不会出问题——问题出在需要 JOIN 三张表、按部门排名、带小计汇总的报表查询上。本文不重复去讲 DDL/DML 基础语法，聚焦于从能查到查好的进阶技能。

一、JOIN：不只是 INNER JOIN ​

1.1 JOIN 类型的可视化理解 ​

INNER JOIN:     返回两张表的交集
LEFT JOIN:      返回左表全部 + 右表匹配（没匹配的填充 NULL）
RIGHT JOIN:     返回右表全部 + 左表匹配（用 LEFT JOIN 改写即可，语义更清晰）
FULL OUTER JOIN: 返回两表的并集（MySQL 不直接支持，用 LEFT JOIN UNION RIGHT JOIN 模拟）

1.2 JOIN 与子查询：何时选哪个？ ​

sql
-- 场景：查询每个用户的最新订单

-- 方案一：子查询 + JOIN（通常更高效）
SELECT u.username, o.order_date, o.amount
FROM users u
JOIN orders o ON o.user_id = u.id
WHERE o.id IN (
    SELECT MAX(id) FROM orders GROUP BY user_id
);

-- 方案二：窗口函数（MySQL 8+，最推荐）
SELECT username, order_date, amount
FROM (
    SELECT u.username, o.order_date, o.amount,
           ROW_NUMBER() OVER (PARTITION BY u.id ORDER BY o.id DESC) AS rn
    FROM users u
    JOIN orders o ON o.user_id = u.id
) t
WHERE rn = 1;

选择法则：

  • 子查询适合 存在性检查（WHERE EXISTS (SELECT 1 FROM ... WHERE ...)），数据库可以在找到第一条匹配后立即停止搜索
  • JOIN 适合 需要多表字段 的场景，可以利用索引做关联
  • 窗口函数适合 分组排名/累计/移动平均，比子查询更清晰且通常更高效

二、窗口函数：分组计算的利器 ​

窗口函数与传统 GROUP BY 的区别：GROUP BY 将多行折叠为一行，丢失了行级别的细节。窗口函数保持了所有行的同时，为每行计算一个基于「窗口」（一组相关行）的值。

sql
-- ROW_NUMBER：为每个部门的员工按薪资排名
SELECT department, name, salary,
       ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) AS rank
FROM employees;

-- RANK vs DENSE_RANK：处理并列
-- salary 100, 100, 90:
--   RANK:       1, 1, 3  (有间隔)
--   DENSE_RANK: 1, 1, 2  (无间隔)

-- LAG / LEAD：与前/后一行比较
SELECT product_id, sale_date, daily_sales,
       LAG(daily_sales, 1) OVER (PARTITION BY product_id ORDER BY sale_date) AS prev_day_sales,
       daily_sales - LAG(daily_sales, 1) OVER (...) AS change_from_prev
FROM daily_sales;

-- 累计求和（Running Total）
SELECT sale_date, amount,
       SUM(amount) OVER (ORDER BY sale_date ROWS UNBOUNDED PRECEDING) AS cumulative
FROM sales;

CTE（公共表表达式）：让复杂查询可读 ​

sql
-- ❌ 嵌套子查询：可读性极差
SELECT ... FROM (
    SELECT ... FROM (
        SELECT ... FROM ...
    ) t1 JOIN (
        SELECT ... FROM ...
    ) t2 ON ...
) t3 WHERE ...;

-- ✅ CTE：从上到下，逐步构建
WITH
user_stats AS (
    SELECT user_id, COUNT(*) AS order_count, SUM(amount) AS total_spent
    FROM orders WHERE status = 'COMPLETED'
    GROUP BY user_id
),
ranked_users AS (
    SELECT user_id, order_count, total_spent,
           RANK() OVER (ORDER BY total_spent DESC) AS spending_rank
    FROM user_stats
)
SELECT u.username, r.order_count, r.total_spent, r.spending_rank
FROM ranked_users r
JOIN users u ON u.id = r.user_id
WHERE r.spending_rank <= 10;

CTE 不改变执行计划——数据库优化器将其内联到主查询中。但它的线性结构让复杂查询从不可维护变为可维护。

三、索引与 EXPLAIN：查询优化的基本功 ​

3.1 EXPLAIN 的关键输出 ​

sql
EXPLAIN SELECT * FROM orders
WHERE user_id = 123 AND status = 'PAID';
输出字段含义关注值
type访问类型（越靠前越好）ALL（全表扫描）→ index → range → ref → const
key实际使用的索引NULL = 没有走索引
rows预估扫描行数与总行数对比，差距大时要优化
Extra额外信息Using filesort（需要排序优化）、Using temporary（需要临时表优化）、Using index（覆盖索引，最好）

3.2 覆盖索引：最好的访问模式 ​

sql
-- 组合索引
CREATE INDEX idx_orders_user_status ON orders(user_id, status, created_at);

-- 覆盖索引查询：SELECT 的字段全在索引中，不需要回表
SELECT user_id, status, created_at
FROM orders
WHERE user_id = 123 AND status = 'PAID';
-- Extra: Using index ← 最优

-- 非覆盖查询：SELECT 了索引中没有的字段，需要回表
SELECT user_id, status, created_at, amount
FROM orders
WHERE user_id = 123 AND status = 'PAID';
-- Extra: NULL ← 需要回表查 amount

覆盖索引意味着查询所需的所有数据在索引中就能找到，不需要再访问聚簇索引（主键索引）获取完整行数据。这是 MySQL 查询优化的最高效模式。

3.3 最左前缀原则 ​

组合索引 (A, B, C) 可以用于以下查询模式：

sql
-- ✅ 可以使用索引
WHERE A = ?                          -- 使用 A
WHERE A = ? AND B = ?                -- 使用 A, B
WHERE A = ? AND B = ? AND C = ?      -- 使用全部三个字段

-- ❌ 不能使用索引（跳过了 A）
WHERE B = ?                          -- 索引失效
WHERE C = ?                          -- 索引失效
WHERE B = ? AND C = ?                -- 索引失效

-- ⚠️ 部分使用索引
WHERE A = ? AND C = ?                -- 只用 A（B 缺失，C 无法参与）

这是组合索引设计中最关键的约束——等值查询条件的前缀连续性决定了索引利用率。设计组合索引时，将等值查询中最常用的字段放在最左边。

四、常见查询优化模式 ​

问题原因优化方案
WHERE YEAR(created_at) = 2025函数包裹列 → 索引失效改为 WHERE created_at BETWEEN '2025-01-01' AND '2025-12-31'
WHERE name LIKE '%keyword%'前导通配符 → 索引失效用全文索引（FULLTEXT）替代
SELECT * 但只需要 3 个字段回表开销 + 网络传输只 SELECT 需要的字段，考虑覆盖索引
LIMIT 100000, 20跳过 10 万行，每条都要读取用游标分页（WHERE id > last_id LIMIT 20）
OR 条件跨多个索引优化器可能选全表扫描改用 UNION ALL 拆分查询

五、总结 ​

概念一句话
JOIN vs 子查询需要多表字段用 JOIN，存在性检查用 EXISTS，分组排名用窗口函数
窗口函数GROUP BY 折叠行，窗口函数保持行 + 附加计算值
EXPLAINtype=ALL 是全表扫描，Extra=Using index 是覆盖索引——最优
覆盖索引SELECT 的字段必须在索引中——避免回表
最左前缀组合索引 (A,B,C) → WHERE 必须以 A 开头才能用到索引

SQL 的进阶不在于记住更多语法，而在于理解数据库是如何执行你的查询的。EXPLAIN 是你与 MySQL 优化器之间的对话界面——学会读它，你做出的优化决策才有根据。
