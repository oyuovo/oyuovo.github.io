引言 ​

大多数 Redis 教程从「安装 → 五大数据结构的命令 → Jedis/Spring Data Redis」这条路径展开。这能让你用上 Redis，但不够理解它。当生产环境出现「这个 Key 为什么占这么多内存」「RDB 持久化怎么把服务卡住了」「集群模式下这个命令为什么报错」时，你需要的是内部机制层面的理解。

本文假设你已能用 SET/GET/HSET 等基本命令。关于缓存穿透、雪崩、击穿等应用层问题，见本博客的 缓存问题 文章。

一、不只是五种数据结构：底层编码的秘密 ​

Redis 对外暴露五种基本数据类型（String、Hash、List、Set、Sorted Set），但底层每种类型对应一到多种内部编码。Redis 根据数据的大小和元素数量自动切换编码，以平衡内存占用和操作效率。

1.1 String 的三种编码 ​

int     → 存整数，值在 long 范围内    → 可直接做 INCR（不需要字符串→整数转换）
embstr  → 存短字符串（≤44 字节）      → 一次内存分配，RedisObject + SDS 连续存储
raw     → 存长字符串（>44 字节）       → 两次内存分配，RedisObject 和 SDS 分开存储
bash
SET counter 100
OBJECT ENCODING counter   # → "int"

SET name "short"
OBJECT ENCODING name      # → "embstr"（≤44 字节）

SET longtext "a very long string..."
OBJECT ENCODING longtext  # → "raw"（>44 字节）

1.2 Hash 的编码切换 ​

ziplist → 元素少且每个 value 短 → 压缩列表（连续内存，省内存但 O(n) 操作）
listpack → Redis 7.0+ 替代 ziplist → 同样是连续内存，解决了级联更新问题
hashtable → 元素多或 value 长 → 标准哈希表（O(1) 操作但内存开销更大）
bash
# 小 Hash：使用 listpack（Redis 7.0+）
HSET user:1 name "Alice" age "25"
OBJECT ENCODING user:1    # → "listpack"

# 大 Hash：超过阈值后自动转为 hashtable
# 阈值由 hash-max-listpack-entries（默认 512）和 hash-max-listpack-value（默认 64 字节）控制

1.3 Sorted Set：跳表的妙用 ​

Sorted Set 同时使用两种数据结构：

  • 跳表（Skiplist）：按 score 排序，支持范围查询（ZRANGEBYSCORE）
  • 哈希表：按 member 查找 score（ZSCORE）

跳表是一种概率平衡的数据结构——通过随机化的层数（level）来保证平均 O(log n) 的查询效率，避免了平衡树的复杂旋转操作：

Level 3:  ┌─head──┐                 ┌─tail──┐
Level 2:  ├─head──┤───────┬─────────┤       │
Level 1:  ├─head──┤───┬───┤───┬─────┤       │
          │  1: A │   │   │   │     │       │
Level 0:  ├─head──┤───┤───┤───┤─────┤       │
          │  "apple":10 │   │     │       │
          │              "banana":20        │
          │                  "cherry":30    │

理解底层编码的实际意义：当你用 HGETALL 遍历一个巨大的 Hash 时，如果它已经从 ziplist/listpack 升级到了 hashtable，时间复杂度从 O(n) 变为更可预测的 O(1) 每次操作——但这同时也意味着更高的内存占用。内存与效率的权衡由 Redis 的编码阈值参数控制，你需要根据业务数据特征进行调整。

二、持久化：RDB、AOF 与混合持久化 ​

Redis 是内存数据库，数据在断电后会丢失。持久化机制将数据写入磁盘，在重启时恢复。

2.1 RDB（快照） ​

RDB 在某个时间点对整个数据集做全量快照：

触发方式：
  - save（主线程执行，阻塞所有请求——生产禁用）
  - bgsave（fork 子进程执行，不阻塞——推荐）
  - 配置自动触发：save 900 1（900 秒内 1 次修改）

执行过程：
  父进程 fork() → 子进程 → 子进程将数据写入临时 RDB 文件
            │                 （利用 Copy-on-Write，此时父进程继续处理请求）
            └─ 继续处理请求
                        → 子进程完成 → 原子替换旧 RDB 文件

优点：文件紧凑，恢复速度快（直接加载到内存）
缺点：两次快照之间的数据可能丢失（取决于快照频率）；fork 大内存进程时可能阻塞（Copy-on-Write 的页表复制开销）

2.2 AOF（追加日志） ​

AOF 将每一条写命令追加到日志文件末尾：

┌─────────────────────────────────────────────┐
│ AOF 文件内容（RESP 协议格式）                   │
├─────────────────────────────────────────────┤
│ *3          ← 3 个参数                       │
│ $3          ← 第一个参数长度                   │
│ SET                                          │
│ $4          ← 第二个参数长度                   │
│ name                                         │
│ $5          ← 第三个参数长度                   │
│ Alice                                        │
└─────────────────────────────────────────────┘

fsync 策略（appendfsync 配置）：

策略行为数据安全性性能
always每条命令都 fsync最高，不丢数据差（磁盘 I/O 瓶颈）
everysec每秒 fsync 一次最多丢 1 秒数据好（推荐）
no由操作系统决定何时 fsync不可控最好（但不推荐）

AOF 重写：随运行时间增长，AOF 文件会膨胀（同一个 Key 被反复修改，日志中记录了大量冗余命令）。Redis 通过 BGREWRITEAOF 在后台重写 AOF 文件——分析当前数据集的状态，为每个 Key 生成最少的命令，大幅压缩文件体积。

2.3 混合持久化（Redis 4.0+） ​

混合持久化的 AOF 文件结构：
┌─────────────┬──────────────────────────┐
│  RDB 格式    │  AOF 格式（增量部分）       │
│  (当前快照)   │  (快照之后的写命令)         │
└─────────────┴──────────────────────────┘
  • 重写 AOF 时，前半部分用 RDB 格式写入当前快照（恢复快），后半部分用 AOF 格式写入增量命令（数据完整）
  • 结合了 RDB 的快速恢复和 AOF 的数据完整性

生产环境推荐配置

properties
# RDB：允许丢失几分钟数据
save 900 1
save 300 10
save 60 10000

# AOF：至少每秒持久化
appendonly yes
appendfsync everysec

# 启用混合持久化
aof-use-rdb-preamble yes

三、内存淘汰策略 ​

当 Redis 内存使用达到 maxmemory 限制时，需要选择淘汰哪些 Key：

策略淘汰对象适用场景
noeviction不淘汰，写请求直接报错不推荐（除非 Redis 只做缓存层，上游有降级方案）
allkeys-lru所有 Key 中最近最少使用的通用缓存场景首选
allkeys-lfu所有 Key 中使用频率最低的热点数据分散的场景
volatile-lru设有 TTL 的 Key 中 LRU持久数据 + 缓存数据混用
volatile-ttl设有 TTL 的 Key 中 TTL 最短的偏向淘汰即将过期的数据
allkeys-random随机淘汰访问模式均匀，无冷热区分

LRU vs LFU：LRU 关注「多久没被访问」，LFU 关注「被访问过多少次」。访问频率高但上次访问较早的数据，LRU 会误淘汰，LFU 不会。

properties
# 推荐配置
maxmemory 4gb                     # 务必设置上限，不能依赖操作系统 OOM Killer
maxmemory-policy allkeys-lru      # 大多数缓存场景的最佳选择

四、Redis Cluster：哈希槽与重定向 ​

当数据量超过单机内存或吞吐量超过单机处理能力时，需要集群。

4.1 哈希槽（Hash Slot） ​

Redis Cluster 将整个键空间划分为 16384 个哈希槽：

slot = CRC16(key) % 16384

每个节点负责一部分槽：

┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│  Node A      │  │  Node B      │  │  Node C      │
│  slot 0-5460 │  │  slot 5461-  │  │  slot 10923- │
│              │  │     10922    │  │     16383    │
└─────────────┘  └─────────────┘  └─────────────┘

4.2 MOVED 与 ASK 重定向 ​

当客户端向错误的节点发送命令时：

客户端 → Node A: GET user:1234
            │ CRC16("user:1234") % 16384 = 7800
            │ 槽 7800 属于 Node B，不在这里
            │
            ← MOVED 7800 <Node B 的 IP:Port>
            │
客户端 → 更新自己的槽位映射缓存
客户端 → Node B: GET user:1234
            ← "Alice" ✅
  • MOVED：槽已经永久迁移到另一个节点，更新缓存后重试
  • ASK：槽正在迁移中（部分 Key 在新节点，部分在旧节点），仅本次重定向，不更新缓存

4.3 集群的限制 ​

Redis Cluster 不支持跨槽多 Key 操作

MSET user:1:name Alice user:2:name Bob ——如果 user:1 和 user:2 落在不同的槽，这个命令会报错。解决方案：使用 Hash Tag 强制相关 Key 落在同一个槽：

MSET {user:1}:name Alice {user:1}:age 25   # {} 内的部分用于计算 slot

Hash Tag 保证 {user:1}:* 的所有 Key 落在同一个节点，但也要注意不要导致数据倾斜（热点 Key 全在同一个节点）。

五、Spring Data Redis 最佳实践 ​

5.1 序列化选择 ​

Spring Data Redis 默认使用 JDK 序列化，这带来两个问题：可读性差（二进制乱码）和内存占用大。技术选型时的决策：

java
@Configuration
public class RedisConfig {

    @Bean
    public RedisTemplate<String, Object> redisTemplate(
            RedisConnectionFactory connectionFactory) {
        RedisTemplate<String, Object> template = new RedisTemplate<>();
        template.setConnectionFactory(connectionFactory);

        // Key 统一用 String 序列化（可读 + 节省空间）
        template.setKeySerializer(RedisSerializer.string());
        template.setHashKeySerializer(RedisSerializer.string());

        // Value 用 JSON 序列化（可读 + 无需手动序列化）
        GenericJackson2JsonRedisSerializer jsonSerializer =
            new GenericJackson2JsonRedisSerializer();
        template.setValueSerializer(jsonSerializer);
        template.setHashValueSerializer(jsonSerializer);

        return template;
    }
}
GenericJackson2JsonRedisSerializer 的一个坑

这个序列化器会在 JSON 中添加 @class 字段存储 Java 类型信息，用于反序列化时自动恢复类型。这会带来约 30-50% 的额外内存开销。

如果不需要类型自动恢复（大多数场景），推荐使用 StringRedisTemplate + 手动 JSON.toJSONString() / JSON.parseObject()，完全控制序列化且零类型开销：

java
@Autowired
private StringRedisTemplate redisTemplate;

// 写
redisTemplate.opsForValue().set("user:1", JSON.toJSONString(user), Duration.ofHours(1));
// 读
User user = JSON.parseObject(redisTemplate.opsForValue().get("user:1"), User.class);

5.2 客户端选择：Jedis vs Lettuce ​

Spring Boot 2.x+ 默认使用 Lettuce：

维度JedisLettuce
连接方式连接池（BIO）单连接复用（NIO, Netty）
线程安全非线程安全，需要连接池原生线程安全
同步/异步/响应式仅同步全部支持
适合场景简单应用，团队熟悉高并发、响应式应用

新项目默认选 Lettuce 即可——Spring Boot 的自动配置已经做了这个选择。

六、总结 ​

层次关键概念一句话
数据结构内部编码（int/embstr/raw, ziplist/listpack/hashtable）同一个 SET 背后可能是完全不同的内存布局
持久化RDB（快照）+ AOF（日志）+ 混合模式everysec + 混合持久化 = 生产标配
淘汰LRU/LFU/noeviction缓存场景选 allkeys-lru，务必设 maxmemory
集群16384 哈希槽 + MOVED/ASK 重定向跨槽操作用 Hash Tag，注意数据倾斜
客户端Lettuce（NIO, 线程安全）Spring Boot 默认，无需切换

Redis 的上手曲线很低，但深入曲线很陡——理解内部编码、持久化机制和集群拓扑，是在生产环境中用好 Redis 的基本功。
