引言 ​

JVM 是 Java 后端开发者的运行平台。大多数时候你不需要感知它的存在——直到线上出现 OutOfMemoryError、类加载冲突、或者 GC 日志暴增。这些时刻，JVM 内部机制的理解直接决定了排查效率。

本文不与 垃圾回收机制 重复。关于 GC 算法（标记-清除、标记-复制、标记-整理）、垃圾回收器（CMS、G1、ZGC）、GC 调优参数和日志分析，请阅读该文。本文专注于：类加载、运行时数据区、JIT 编译、OOM 场景复现。

二、类加载机制：双亲委派与如何破坏它 ​

2.1 类加载的七个阶段 ​

加载 → 验证 → 准备 → 解析 → 初始化 → 使用 → 卸载
         └─── 链接阶段 ────┘

加载：通过类的全限定名获取二进制字节流 → 将静态存储结构转换为方法区运行时数据结构 → 在堆中生成 java.lang.Class 对象

验证：检查 class 文件格式、元数据、字节码、符号引用是否符合 JVM 规范

准备：为 static 变量在方法区分配内存并设置默认零值（int → 0, Object → null）。注意：此时只分配零值，显式赋值在初始化阶段执行

解析：将常量池中的符号引用（如 com/example/UserService）替换为直接引用（内存地址）

初始化：执行类构造器 <clinit>()——编译器按代码顺序收集 static 变量赋值和 static {} 代码块。父类的 <clinit>() 保证在子类之前执行

2.2 双亲委派模型 ​

java
// 类加载器层次结构
Bootstrap ClassLoader           ← 加载 rt.jar（java.lang.*, java.util.* 等）
        ↑
Extension/Platform ClassLoader  ← 加载 jre/lib/ext 或 java.platform 模块
        ↑
Application ClassLoader          ← 加载 classpath 下的应用类
        ↑
Custom ClassLoader               ← 用户自定义（如 Tomcat 的 WebappClassLoader）

双亲委派的核心逻辑（在 ClassLoader.loadClass() 中实现）：

1. 检查当前加载器是否已加载过该类 → 已加载则直接返回
2. 委托父类加载器加载 → 父类成功则返回
3. 父类加载失败 → 当前加载器自己尝试加载（findClass）

为什么需要双亲委派？ 防止核心类库被篡改。如果有人写了一个 java.lang.String 类并放到 classpath 中，Application ClassLoader 不会直接加载它——请求会一路向上到达 Bootstrap ClassLoader，发现 java.lang.String 已被加载，直接返回标准库的版本。

2.3 破坏双亲委派的三种场景 ​

场景一：SPI 机制（JDK 自己的反例） ​

JDBC 的 DriverManager 由 Bootstrap ClassLoader 加载，但它需要调用由 Application ClassLoader 加载的第三方驱动实现类（如 com.mysql.cj.jdbc.Driver）。双亲委派模型下，Bootstrap ClassLoader 加载的类无法「向下」看到子加载器的类。

解决方案：线程上下文类加载器（Thread Context ClassLoader）

java
// DriverManager 内部的加载逻辑（简化）
ServiceLoader<Driver> loader = ServiceLoader.load(Driver.class);
// load() 方法内部：
Thread.currentThread().getContextClassLoader()  // 获取 App ClassLoader
    .loadClass("com.mysql.cj.jdbc.Driver")       // 用 App ClassLoader 加载 SPI 实现

线程上下文类加载器默认是 Application ClassLoader。通过将加载权从双亲委派的「向上」改为「向下」，SPI 接口和实现可以在不同层级的类加载器中并存。

场景二：Tomcat 的 WebappClassLoader ​

Tomcat 需要满足三个特殊需求：

  1. 不同 Web 应用之间的类隔离（com.example.User 在两个应用中可以有不同的版本）
  2. 不同 Web 应用之间的类共享（公共库如 servlet-api 不需要每个应用各加载一份）
  3. Web 应用可以重新部署（热加载）

Tomcat 的做法是每个 Web 应用使用独立的 WebappClassLoader，加载顺序与双亲委派完全相反——先自己尝试加载，加载不到再委托父加载器。

场景三：热部署/热加载 ​

OSGi、DevTools、JRebel 等热部署工具都通过自定义类加载器绕过了双亲委派。它们为每个模块/版本创建独立的类加载器，实现模块级别的动态卸载和替换。

2.4 类卸载的条件 ​

一个类被 GC 卸载需要同时满足三个条件（这三个条件在 JVM 规范中是明确规定的）：

  1. 该类的所有实例已被回收——堆中不存在该类的任何实例对象
  2. 该类的 ClassLoader 已被回收——加载该类的类加载器实例不再存活
  3. 该类的 Class 对象不再被引用——没有任何地方通过反射引用该类的 Class 对象

实际意义：在 Tomcat 中，每次重新部署都会创建新的 WebappClassLoader。如果旧 ClassLoader 因为线程局部变量、静态引用等没有被回收，就会导致元空间泄漏——不断增长的类元数据最终触发 OutOfMemoryError: Metaspace。

三、JIT 编译：从解释执行到机器码 ​

3.1 为什么需要 JIT？ ​

JVM 启动时通过解释器逐条执行字节码——这保证了快速启动，但解释执行比直接执行机器码慢 10-100 倍。JIT（Just-In-Time）编译器在运行时将热点代码编译为本地机器码，后续调用直接执行机器码。

3.2 C1 编译器 vs C2 编译器 ​

HotSpot VM 内建两个 JIT 编译器：

编译器策略特点适用阶段
C1（Client Compiler）快速编译编译速度快，优化程度较低启动初期，GUI 应用
C2（Server Compiler）深度优化编译时间较长，生成的机器码效率高服务端长期运行

3.3 分层编译（Tiered Compilation） ​

Java 7 开始默认启用分层编译，结合 C1 和 C2 两个编译器的优势：

解释执行 (Level 0)
    │
    ▼ (方法被调用达到阈值)
C1 编译，不开启 profiling (Level 1)     ← 快速获得编译机器码
    │
    ▼ (继续被频繁调用)
C1 编译 + 完整 profiling (Level 2)      ← 收集运行时数据
    │
    ▼ (profiling 显示这是超级热点)
C2 编译 + profiling 数据驱动深度优化 (Level 3)   ← 最高性能的机器码

profiling 数据包括：分支概率、方法调用目标、类型分布等。C2 利用这些数据进行内联（将小方法体直接嵌入调用处，消除方法调用开销）、逃逸分析（栈上分配、锁消除）、去虚拟化（确定虚方法的唯一实现目标转为直接调用）。

3.4 查看编译日志 ​

bash
# 打印被 JIT 编译的方法
java -XX:+PrintCompilation -jar app.jar

# 输出格式：
#  287   1       3    java.lang.String::hashCode (55 bytes)
#  │     │       │    │
#  │     │       │    └─ 方法名和字节码大小
#  │     │       └─ 编译层级 (3 = C2)
#  │     └─ 编译 ID（序号）
#  └─ 编译发生时的毫秒时间戳

生产环境中不建议开启 -XX:+PrintCompilation（日志量极大），但当性能分析时，编译日志可以帮助确认热点方法是否被 C2 充分优化。

四、OOM 场景复现与排查 ​

4.1 Heap OOM ​

最常见的 OOM 类型——堆中对象过多且无法被 GC 回收。

java
// -Xmx256m -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/heapdump.hprof
public class HeapOOM {
    public static void main(String[] args) {
        List<byte[]> list = new ArrayList<>();
        while (true) {
            list.add(new byte[10 * 1024 * 1024]);  // 每次分配 10MB
        }
    }
}
// 错误信息: java.lang.OutOfMemoryError: Java heap space

排查思路：-XX:+HeapDumpOnOutOfMemoryError 自动导出堆转储 → 用 MAT/JProfiler/Eclipse Memory Analyzer 分析 → 找到占用内存最大的对象及其 GC Root 引用链。

4.2 Metaspace OOM ​

元空间的容量由类元数据消费决定，与堆无关。当动态生成大量类（CGLIB 代理、Groovy/JSP 编译、大量 Lambda）时触发。

java
// -XX:MaxMetaspaceSize=64m
public class MetaspaceOOM {
    public static void main(String[] args) {
        while (true) {
            Enhancer enhancer = new Enhancer();
            enhancer.setSuperclass(OOMObject.class);
            enhancer.setUseCache(false);
            enhancer.setCallback((MethodInterceptor) (obj, method, args1, proxy) ->
                proxy.invokeSuper(obj, args1));
            enhancer.create();  // CGLIB 每次生成一个新类
        }
    }
    static class OOMObject {}
}
// 错误信息: java.lang.OutOfMemoryError: Metaspace

Metaspace vs PermGen

Java 8 用 Metaspace 替代了 PermGen（永久代）。关键变化：PermGen 大小由 -XX:MaxPermSize 限制，Metaspace 默认使用本地内存，上限受限于操作系统可用内存——如果不设置 -XX:MaxMetaspaceSize，理论上可以耗尽所有本地内存。生产环境务必设置 -XX:MaxMetaspaceSize，推荐值 256m-512m。

4.3 线程数 OOM ​

每个线程需要独立的栈空间（默认 ~1MB），大量线程占用的是本地内存而非堆内存。

java
// -Xss256k（缩小栈以容纳更多线程）
public class ThreadOOM {
    public static void main(String[] args) {
        while (true) {
            new Thread(() -> {
                try { Thread.sleep(Long.MAX_VALUE); }  // 永远不退出
                catch (InterruptedException e) {}
            }).start();
        }
    }
}
// 错误信息: java.lang.OutOfMemoryError: unable to create native thread

排查: jstack <pid> 查看线程状态 → 如果大量线程处于 WAITING/TIMED_WAITING 状态 → 检查线程池是否创建了过多的线程或线程泄漏（每次请求创建新线程但从不回收）。

4.4 直接内存 OOM ​

NIO 的 ByteBuffer.allocateDirect() 分配的是堆外内存，不受 -Xmx 限制。

java
// -XX:MaxDirectMemorySize=256m
public class DirectMemoryOOM {
    public static void main(String[] args) {
        List<ByteBuffer> list = new ArrayList<>();
        while (true) {
            list.add(ByteBuffer.allocateDirect(10 * 1024 * 1024));  // 堆外 10MB
        }
    }
}
// 错误信息: java.lang.OutOfMemoryError: Direct buffer memory

排查: 堆转储中看不到直接内存内容 → 检查 jcmd <pid> VM.native_memory summary → 关注 Internal 区域的 Direct Byte Buffer 项。

五、总结 ​

机制核心要点排障关键
运行时数据区堆 + 方法区/元空间 + 虚拟机栈，各区域职责分明OOM 时先判断哪个区域 → 错误信息直接指明堆/元空间
双亲委派自底向上委派加载，保证核心类库安全ClassNotFoundException → 检查类加载器层级和 TCCL
JIT 编译C1 快速编译 + C2 深度优化 = 分层编译用 -XX:+PrintCompilation 验证热点方法被优化
OOM 排查4 种 OOM 类型，每种有特定的原因和解决方案-XX:+HeapDumpOnOutOfMemoryError + MAT 分析

JVM 的内部机制与垃圾回收共同定义了 Java 的运行时行为。掌握其中任何一半，排查性能问题和内存故障时思路都会完全不同。
