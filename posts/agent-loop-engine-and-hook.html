Agent Loop Engine 与 Hook：从运行机制到无人监督编码 ​

1. Agent 系统中的三层循环 ​

Agent 相关系统中经常混用几个相似概念：

概念关注范围主要职责
Agent Loop单次任务中的模型行为观察、决策、行动、再观察
Loop Engine单个 Agent 的运行时调度、状态、工具、重试、终止
Loop Engineering整个自动化工作系统任务发现、分配、验证、返工、审批

三者可以表示为：

text
Loop Engineering
└── 管理完整任务生命周期
    └── Loop Engine
        └── 管理一次 Agent 运行
            └── Agent Loop
                └── 模型、工具、观察、决策

Agent Loop ​

Agent Loop 是智能体最基础的运行结构：

text
读取当前状态
→ 决定下一步动作
→ 执行动作
→ 获取新结果
→ 更新状态
→ 判断继续或结束

简化成代码：

python
while not finished:
    action = model.decide(state)
    result = execute(action)
    state = update_state(state, action, result)
    finished = check_finished(state)

Loop 的意义不在于循环次数多，而在于：

下一步操作依赖上一步结果，执行路径无法在任务开始前完全确定。

一个 Agent 可能只运行两三轮，但只要后续行为由中间结果决定，它就处于 Agent Loop 中。

Loop Engine ​

Loop Engine 是承载 Agent Loop 的运行时系统。

模型负责提出下一步动作，Loop Engine 负责决定：

  • 动作是否允许执行；
  • 应该调用哪个工具；
  • 工具结果如何写入状态；
  • 失败时是否重试；
  • 是否已经达到目标；
  • 是否应该立即停止。

Loop Engineering ​

Loop Engineering 关注的范围更大。它不仅管理一个 Agent 会话，还管理多个任务之间的流转：

text
发现任务
→ 创建工作区
→ 启动 Agent
→ 执行验证
→ 失败返工
→ 生成 PR
→ 人工审批
→ 结束或进入下一任务

传统 Vibe Coding 主要发生在 Agent Loop 内部，而长期无人监督 Coding 需要 Loop Engineering。

2. Loop Engine 的内涵 ​

Loop Engine 不能简单理解为一个 while 循环。

一个真正有工程价值的 Loop Engine，本质上是 Agent 的控制平面。

模型只负责生成候选决策：

text
读取某个文件
修改某段代码
执行某条命令
搜索某项信息
结束任务

Loop Engine 负责把候选决策转化为受约束的真实操作：

text
模型提出动作
→ 验证动作格式
→ 检查权限
→ 检查预算
→ 执行工具
→ 记录结果
→ 更新状态
→ 运行终止判断

因此，Loop Engine 的核心不是“让 Agent 继续运行”，而是：

让 Agent 只能在预先定义的边界内继续运行。

Loop Engine 的输入 ​

典型输入包括：

  • 用户目标；
  • 系统规则；
  • 可用工具；
  • 当前状态；
  • 历史执行轨迹；
  • 权限配置；
  • 时间和成本预算；
  • 验收条件。

Loop Engine 的输出 ​

典型输出包括：

  • 最终答案；
  • 修改后的代码；
  • Patch 或 Pull Request；
  • 任务失败报告；
  • 检查点；
  • 工具执行日志；
  • 明确的停止原因。

Loop Engine 的运行模型 ​

一个较完整的 Loop Engine 可以表示为：

python
def run_agent(task):
    state = initialize_state(task)

    for step in range(task.max_steps):
        run_hooks("before_step", state)

        action = call_model(state)
        run_hooks("after_model_call", state, action)

        validate_action(action)
        enforce_policy(action)

        result = execute_action(action)
        run_hooks("after_tool_call", state, action, result)

        state = update_state(state, action, result)
        save_checkpoint(state)

        decision = evaluate_stop_conditions(state)

        if decision.should_stop:
            run_hooks("before_stop", state, decision)
            return finalize(state, decision)

    return finalize(state, reason="max_steps_reached")

这里的循环只是外壳，真正有价值的是循环内部的控制结构。

3. Loop Engine 的核心组成 ​

一个完整的 Loop Engine 通常由六部分组成。

3.1 Planner：动作决策 ​

Planner 根据当前状态决定下一步动作。

它可能由大语言模型实现：

text
当前目标：修复登录失败
当前发现：Token 过期时间单位错误
下一步动作：修改 token.ts

Planner 不应该直接拥有系统权限。它只生成结构化动作：

json
{
  "type": "edit_file",
  "path": "src/auth/token.ts",
  "changes": "..."
}

这样可以把“决定做什么”和“真正执行”分开。

3.2 Executor：工具执行 ​

Executor 把动作转换成真实操作：

  • 读取文件；
  • 修改代码；
  • 运行命令；
  • 搜索仓库；
  • 调用浏览器；
  • 运行测试；
  • 调用外部 API。

Executor 是风险最集中的部分，因为这里会真正改变外部环境。

可靠的 Executor 应具备：

  • 参数校验；
  • 超时控制；
  • 输出大小限制；
  • 文件路径限制；
  • 命令白名单或黑名单；
  • 沙箱；
  • 错误分类。

3.3 State Store：状态管理 ​

State Store 保存 Agent 当前掌握的信息。

典型状态包括：

python
state = {
    "goal": "修复登录失败问题",
    "step": 6,
    "completed_actions": [
        "读取 token.ts",
        "修改过期时间计算"
    ],
    "observations": [
        "单位由秒误写为毫秒",
        "仍有两个刷新 Token 测试失败"
    ],
    "failed_attempts": [
        "仅修改 access token 没有解决 refresh token"
    ],
    "current_diff": "...",
    "remaining_budget": 12
}

状态管理可以防止：

  • 重复读取同一文件；
  • 重复尝试已经失败的方案；
  • 长任务中丢失原始目标；
  • 上下文过长导致关键信息被忽略；
  • 进程中断后无法恢复。

3.4 Policy Layer：权限与安全 ​

Policy Layer 决定 Agent 可以做什么。

例如：

text
允许：
- 读取 src/**
- 修改 tests/**
- 运行 npm test

禁止：
- 读取 .env
- 访问 ~/.ssh
- 执行 git push
- 修改 main 分支
- 执行生产部署

策略检查通常发生在工具执行之前：

python
def enforce_policy(action):
    if action.path.startswith(".env"):
        raise PolicyViolation("禁止读取秘密文件")

    if action.command.startswith("git push"):
        raise PolicyViolation("禁止推送远程分支")

模型是否“理解了规则”不能作为安全保障。

可靠的权限必须由模型之外的代码实施。

3.5 Evaluator：结果验证 ​

Evaluator 判断当前结果是否满足任务目标。

模型可能会说：

text
代码已经修复，任务完成。

但系统真正关心的是：

text
测试是否通过？
lint 是否通过？
构建是否成功？
是否修改了禁止文件？
是否引入了新的失败？

因此，完成判断应尽量依赖可执行条件：

bash
npm run lint
npm run typecheck
npm test
npm run build

Evaluator 的核心原则是：

Agent 可以提出“我认为完成了”，但最终完成状态必须由独立证据确认。

3.6 Stop Controller：终止控制 ​

Stop Controller 统一处理所有停止条件。

成功停止 ​

text
测试全部通过
验收脚本退出码为 0
目标文件成功生成
用户要求已经满足

资源停止 ​

text
达到最大轮数
超过最长运行时间
超过 Token 预算
超过金额预算

无进展停止 ​

text
连续多轮没有代码变化
相同测试连续失败
重复调用相同工具和参数
没有获得新的有效信息

安全停止 ​

text
请求读取秘密文件
请求部署生产环境
请求删除高价值资源
请求绕过沙箱
请求修改未授权目录

外部停止 ​

text
人工取消任务
CI Runner 被关闭
任务优先级发生变化
上游依赖已经失效

Stop Controller 应输出明确原因：

json
{
  "status": "stopped",
  "reason": "no_progress",
  "step": 14,
  "last_error": "refresh token test failed three times"
}

4. Hook 在 Loop 中的位置 ​

Hook 是 Loop Engine 在关键执行节点暴露的扩展接口。

它允许外部逻辑在不修改主循环代码的情况下，观察、拦截或补充执行过程。

典型执行流程：

text
开始任务
│
├── before_agent_start
│
├── 开始新一轮
│   ├── before_step
│   ├── before_model_call
│   ├── 调用模型
│   ├── after_model_call
│   ├── before_tool_call
│   ├── 执行工具
│   ├── after_tool_call
│   ├── 更新状态
│   └── after_step
│
├── 判断是否结束
│   ├── before_finish
│   └── after_finish
│
└── 出错时触发 on_error

Hook、Loop Engine 与业务逻辑的关系 ​

text
Loop Engine
├── 定义主流程
├── 控制状态和终止
└── 暴露 Hook 点
    ├── 日志 Hook
    ├── 权限 Hook
    ├── 验证 Hook
    ├── 监控 Hook
    └── 检查点 Hook

Loop Engine 决定流程的骨架，Hook 负责在骨架上的固定节点附加能力。

5. Hook 的典型使用方式 ​

5.1 日志 Hook ​

日志 Hook 用于记录每轮的输入、动作和结果。

python
def log_before_tool(context):
    logger.info({
        "step": context.step,
        "tool": context.tool_name,
        "arguments": context.arguments
    })

engine.add_hook("before_tool_call", log_before_tool)

适合记录：

  • 工具名称；
  • 输入参数；
  • 当前轮数；
  • 当前目标；
  • 工具运行时间；
  • 执行结果；
  • 错误信息。

5.2 权限 Hook ​

权限 Hook 在真实操作发生前拦截危险动作。

python
def block_dangerous_command(context):
    command = context.arguments.get("command", "")

    forbidden = [
        "git push",
        "rm -rf",
        "kubectl delete",
        "terraform destroy"
    ]

    if any(item in command for item in forbidden):
        raise PermissionError(
            f"禁止执行命令：{command}"
        )

这种 Hook 最适合实现横跨多个工具的安全策略。

5.3 参数标准化 Hook ​

模型生成的参数可能过于宽泛，Hook 可以在执行前进行收缩。

python
def restrict_timeout(context):
    requested = context.arguments.get("timeout", 60)
    context.arguments["timeout"] = min(requested, 300)

还可以用于：

  • 限制搜索结果数量；
  • 限制命令运行时间；
  • 规范文件路径；
  • 清理非法参数；
  • 强制添加只读标志。

5.4 检查点 Hook ​

长期任务需要在关键阶段保存状态：

python
def save_checkpoint(context):
    checkpoint_store.save(
        task_id=context.task_id,
        step=context.step,
        state=context.state
    )

engine.add_hook("after_step", save_checkpoint)

检查点通常保存：

  • 当前目标；
  • 已完成步骤；
  • 当前代码差异；
  • 已知错误；
  • 剩余预算；
  • 最近工具结果；
  • 当前工作区位置。

5.5 验证 Hook ​

验证 Hook 可以防止 Agent 过早结束。

python
def verify_before_finish(context):
    result = run_command(
        "./scripts/verify-agent.sh"
    )

    if result.exit_code != 0:
        raise ContinueLoop(
            feedback=result.stderr
        )

典型流程：

text
Agent 请求结束
→ before_finish Hook 运行测试
→ 测试失败
→ 阻止结束
→ 将失败信息返回 Agent
→ Agent 继续修改

5.6 监控 Hook ​

监控 Hook 用于采集系统指标：

text
每轮 Token 消耗
模型调用延迟
工具执行时间
测试失败次数
文件修改数量
任务总成本
重复动作次数

这些指标可以用于识别：

  • Agent 是否陷入循环；
  • 某个工具是否过慢；
  • 某类任务是否成本过高；
  • 某个模型是否经常选错工具。

5.7 Hook 的使用边界 ​

Hook 适合承担横切逻辑：

text
日志
监控
权限
审计
检查点
参数规范化
通用验证

不适合把整个业务流程隐藏在 Hook 中。

不良设计：

text
before_step 决定任务类型
after_model_call 修改核心目标
before_tool_call 重写整个计划
after_tool_call 决定下一阶段

当主要逻辑分散在多个 Hook 中时，会出现：

  • 执行顺序难以理解；
  • 参数不知道在哪里被修改；
  • 多个 Hook 相互覆盖；
  • 故障难以重现；
  • 主流程看起来简单，实际行为却很复杂。

核心流程应显式存在于 Loop Engine 或状态机中，Hook 只承担附加能力。

6. Loop Engine 的可观测性设计 ​

Loop Engine 会动态选择执行路径，因此调试重点不是只看最终结果，而是还原整条执行轨迹。

一条完整轨迹应至少包含：

text
Step 1
类型：模型决策
输入：修复登录测试
输出：读取 src/auth/token.ts

Step 2
类型：工具调用
工具：read_file
参数：src/auth/token.ts
结果：读取成功

Step 3
类型：模型决策
结论：过期时间单位错误
动作：修改 token.ts

Step 4
类型：工具调用
工具：edit_file
结果：修改成功

Step 5
类型：工具调用
工具：run_tests
结果：18 个测试通过，2 个失败

故障分类 ​

完整轨迹可以帮助定位问题属于哪一层：

故障类型典型表现
模型决策错误选错文件、选错工具、方向错误
参数生成错误文件路径错误、命令参数不合法
工具执行错误命令失败、网络异常、进程超时
环境错误依赖缺失、版本不一致、权限不足
状态管理错误重复尝试、遗漏信息、状态覆盖
策略错误正常操作被误拦截
终止错误过早完成或无限继续
验证错误测试范围不足或验收条件错误

每一步建议记录的字段 ​

json
{
  "task_id": "task-123",
  "step": 5,
  "event": "tool_call",
  "tool": "run_tests",
  "arguments": {
    "target": "tests/auth"
  },
  "duration_ms": 18342,
  "exit_code": 1,
  "state_version": 7,
  "timestamp": "..."
}

可靠的 Loop Engine 应做到：

  • 每一步有唯一编号；
  • 每次状态更新有版本；
  • 每次工具调用有输入和输出；
  • 每次停止有明确原因；
  • 每次修改可以关联到对应决策。

7. Vibe Coding 中隐含的循环 ​

Vibe Coding 从一开始就存在循环，只是过去没有显式命名。

一个常见过程是：

text
用户提出需求
→ Agent 修改代码
→ Agent 运行测试
→ Agent 返回结果
→ 用户检查
→ 用户继续提出修改

这个过程实际上包含两层循环。

内层循环 ​

Agent 在一次会话中执行：

text
读取代码
→ 修改
→ 测试
→ 再修改

这部分由 Codex、Claude Code 等 Coding Agent 完成。

外层循环 ​

用户负责：

text
检查结果
→ 判断是否真的完成
→ 提供下一条指令
→ 阻止错误方向
→ 决定是否继续

传统 Vibe Coding 中，人类实际上就是外层 Loop Engine。

人承担了以下工作：

  • 验收结果；
  • 补充上下文；
  • 发现遗漏；
  • 决定重试；
  • 约束权限；
  • 判断停止；
  • 承担最终风险。

概念近年来受到重视的原因 ​

变化不在于 Agent 最近才学会循环，而在于系统开始尝试自动化外层循环。

过去：

text
Agent 做一轮
→ 人检查
→ 人发下一条提示

现在：

text
Agent 做一轮
→ 自动测试
→ 自动评估
→ 自动生成反馈
→ 自动继续或停止

当人的监督职责开始被程序替代时，Loop Engine 就从隐含机制变成了独立的工程组件。

8. 从 Agent Loop 到长期无人监督运行 ​

长期无人监督 Coding 不应该依赖一个无限持续的会话。

更可靠的方式是把工作拆成多个有限任务：

text
长期目标
├── 任务 A：修复登录测试
├── 任务 B：补充边界测试
├── 任务 C：重构认证模块
└── 任务 D：生成 Pull Request

每个任务都具有：

  • 独立工作区；
  • 明确允许范围；
  • 明确完成条件；
  • 最大运行时间；
  • 最大执行轮数；
  • 最大成本；
  • 独立验收；
  • 明确失败输出。

安全运行模型 ​

text
任务队列
→ 创建临时分支或 worktree
→ 启动 Coding Agent
→ Agent 在沙箱中工作
→ Loop Engine 记录每一步
→ 外部验证脚本验收
→ 成功则生成 Patch 或 PR
→ 失败则保存状态并停止

任务边界示例 ​

text
目标：
修复登录模块中的 Token 过期问题。

允许修改：
src/auth/**
tests/auth/**

完成条件：
- 登录测试全部通过
- lint 通过
- 类型检查通过
- 构建成功

禁止：
- 修改数据库结构
- 删除失败测试
- 读取 .env
- 执行 git push
- 部署生产环境

失败处理：
输出失败原因、当前代码差异和已尝试方案。

9. Codex 与 Claude Code 的外层控制 ​

Codex 和 Claude Code 已经具备内层 Agent Loop。

它们能够：

text
读取文件
→ 修改代码
→ 执行命令
→ 观察结果
→ 继续修复

开发者通常不需要修改它们的内部循环。

真正需要补充的是外层控制系统：

text
外层监督脚本
├── 创建工作区
├── 启动 Agent
├── 设置超时
├── 设置预算
├── 限制权限
├── 保存日志
├── 运行独立验收
└── 生成最终结果

外部超时 ​

不要只在 Prompt 中告诉 Agent：

text
运行 45 分钟后停止。

应由操作系统或 CI 强制执行：

bash
timeout --signal=TERM --kill-after=30s 45m agent-command

独立验收 ​

建立统一验收脚本：

bash
#!/usr/bin/env bash
set -euo pipefail

npm run lint
npm run typecheck
npm test
npm run build

Agent 的最终文本不作为完成依据，验收脚本的退出码才是完成依据。

独立工作区 ​

bash
git worktree add ../agent-task-123 \
  -b agent/task-123

Agent 只在该工作区中修改代码，不直接接触主分支。

高风险操作隔离 ​

以下操作不应由长期无人监督 Agent 直接完成：

text
合并主分支
发布软件包
生产部署
数据库迁移
修改 IAM
删除云资源
使用生产密钥
向真实用户发送消息

推荐边界：

text
Agent 可以生成代码和 PR
人类或受控系统负责合并和部署

10. Loop Engine 的适用边界 ​

并不是所有 AI 功能都需要 Loop Engine。

适合固定工作流的任务 ​

text
翻译
摘要
文本润色
数据格式转换
固定字段提取
简单分类
模板生成

这类任务的路径可以预先确定：

text
输入
→ 模型
→ 输出

增加 Agent Loop 只会带来：

  • 更高延迟；
  • 更高成本；
  • 更多不确定性；
  • 更复杂调试。

适合 Loop Engine 的任务 ​

text
复杂代码修复
跨文件重构
多工具研究
自动化运维
数据分析
长期任务执行
动态信息收集
多阶段验证

它们通常具有以下特征：

  • 后续步骤依赖中间结果；
  • 工具可能失败；
  • 执行路径无法预先写死；
  • 需要重试或更换方案；
  • 需要持续状态；
  • 需要明确安全边界；
  • 需要独立验收。

11. 最小可用架构 ​

一个个人项目中的最小无人监督 Coding 架构可以是：

text
Git worktree
+ Coding Agent
+ 沙箱
+ 最大运行时间
+ 最大执行轮数
+ 独立验证脚本
+ JSONL 日志
+ 禁止 push 和 deploy

运行结构：

text
TASK.md
   │
   ▼
创建 worktree
   │
   ▼
启动 Codex 或 Claude Code
   │
   ▼
Agent 内部循环
   │
   ├── 读取代码
   ├── 修改代码
   ├── 运行测试
   └── 继续修复
   │
   ▼
外层验证脚本
   │
   ├── 成功 → 保存 Patch
   └── 失败 → 保存日志和状态

对于无人监督 Coding，最终目标不是让 Agent 永远运行，而是让它：

在隔离环境中，以有限权限和有限资源完成一个明确任务，通过独立验证后停止；无法完成时保存现场并安全退出。
