引言 ​

Elasticsearch（ES）是目前最流行的开源搜索引擎，基于 Apache Lucene 构建。它不仅用于全文搜索，也是日志分析（ELK Stack）、实时监控、数据聚合分析的通用平台。

对后端开发者来说，ES 的学习曲线有两个陡坡：一是倒排索引这个与关系型数据库完全不同的数据模型，二是查询 DSL 的表达能力远超 SQL 的 WHERE 子句。本文从这两点出发，逐步延伸到 Java 客户端集成和生产环境运维考量。

一、核心概念：倒排索引与文档模型 ​

1.1 MySQL 为什么不适合全文搜索？ ​

MySQL 的 B+Tree 索引擅长精确匹配和范围查询，但全文搜索中你需要的是分词 + 反向查找：

MySQL:  用户输入 "Java教程" → WHERE content LIKE '%Java教程%'
        → 全表扫描（索引失效），上百万行数据 10 秒+

ES:     用户输入 "Java教程" → 分词为 ["Java", "教程"]
        → 从倒排索引中查找含有 "Java" 和 "教程" 的文档
        → 毫秒级返回

1.2 倒排索引的结构 ​

正排索引（MySQL）:  文档ID → 内容
    1 → "Elasticsearch is a search engine"
    2 → "Redis is a cache"

倒排索引（ES）:     词项（Term） → 包含该词的文档ID列表
    "elasticsearch" → [1]
    "is"            → [1, 2]
    "search"        → [1]
    "engine"        → [1]
    "redis"         → [2]
    "cache"         → [2]

查询 "search engine" 时，ES 分词为 ["search", "engine"]，在倒排索引中分别查找，得到两个文档 ID 列表的交集 [1] ∩ [1] = [1]，这就是文档 1。

1.3 文档、索引、映射的三层抽象 ​

ES 概念MySQL 类比说明
文档（Document）行（Row）JSON 格式的数据记录
索引（Index）表（Table）同类文档的集合
映射（Mapping）表结构（Schema）定义字段类型和索引策略

Mapping 中的关键字段类型选择：

text        → 需要分词的全文搜索字段（如文章标题、内容描述）
keyword     → 精确匹配字段（如状态码、标签、ID、邮箱）
              keyword 不会被分词，可用作排序、聚合、过滤
long/integer → 数值型，参与范围查询和排序
date        → 日期型，支持丰富的日期数学表达式（now-7d/d）

text vs keyword 选择法则

用户会输入部分关键词搜索 → text（如「Java教程」匹配包含「Java」和「教程」的文档）。需要精确匹配或聚合统计 → keyword（如按 status: "published" 过滤、按 tag 分组统计）。许多字段两种类型都需要——ES 支持多字段映射（一个字段同时有 text 和 keyword 两个子字段）。

二、DSL 查询：从简单到复合 ​

ES 的查询语言 DSL（Domain Specific Language）是 JSON 格式的请求体。以下覆盖从基本到高级的查询模式。

2.1 全文搜索与精确过滤 ​

json
// GET /products/_search
{
  "query": {
    "bool": {
      "must": [                          // 必须匹配（贡献得分）
        { "match": { "name": "无线蓝牙耳机" }}
      ],
      "filter": [                        // 必须匹配（不贡献得分，可缓存）
        { "term": { "brand": "Sony" }},
        { "range": { "price": { "gte": 100, "lte": 500 }}}
      ]
    }
  }
}

关键区别：

  • must：匹配的文档相关性得分高，适合搜索词的文本匹配
  • filter：二值判断（是/否），ES 会自动缓存 filter 结果，适合状态、分类、价格范围等结构化过滤

2.2 复合查询实战：多条件搜索 + 结果增强 ​

json
{
  "query": {
    "function_score": {
      "query": {
        "bool": {
          "should": [                    // 或逻辑（匹配越多，得分越高）
            { "match": { "name": "耳机" }},
            { "match": { "brand": "Sony" }}
          ],
          "filter": [
            { "term": { "status": "on_sale" }}
          ]
        }
      },
      "functions": [                    // 得分增强
        {
          "filter": { "term": { "isNewProduct": true }},
          "weight": 2                    // 新品加权 2 倍
        },
        {
          "gauss": {                     // 衰减函数：销量越高的得分越高
            "salesCount": {
              "origin": "10000",
              "scale": "5000"
            }
          }
        }
      ],
      "score_mode": "multiply",
      "boost_mode": "multiply"
    }
  },
  "highlight": {
    "fields": {
      "name": {
        "pre_tags": ["<em>"],
        "post_tags": ["</em>"]
      }
    }
  }
}

function_score 允许在文本相关性得分之上叠加业务逻辑——新品加权、销量加权、距离衰减——这些都是搜索引擎在生产中的关键调优手段。

2.3 聚合（Aggregation）：ES 的 GROUP BY ​

json
{
  "size": 0,                            // 不需要原始文档，只要聚合结果
  "aggs": {
    "by_brand": {
      "terms": { "field": "brand", "size": 20 }
    },
    "avg_price_by_brand": {
      "terms": { "field": "brand" },
      "aggs": {
        "avg_price": { "avg": { "field": "price" }}
      }
    },
    "price_histogram": {
      "histogram": { "field": "price", "interval": 100 }
    }
  }
}

聚合操作在数据量大的时候非常消耗内存。永远设置 size 参数限制返回的桶数量——默认值是 10，但设置为 0 意味着 ES 将返回全部桶，可能触发内存溢出。

三、分页策略：三种方案的选择 ​

方案原理适用场景限制
from + size跳过 from 条，取 size 条浅分页（前 100 页内）from+size > 10000 时性能急剧下降
scroll生成数据快照，游标遍历全量数据导出占用资源，不能实时反映变更
search_after用上一页的最后一条数据的排序值作为下一页起点深分页、无限滚动必须搭配排序字段，不能跳页
json
// search_after —— 推荐的深分页方案
// 第 1 页
{
  "size": 20,
  "sort": [{ "price": "desc" }, { "_id": "desc" }],
  "query": { "match_all": {} }
}
// 拿到最后一条的 sort 值: [1999, "abc123"]

// 第 2 页
{
  "size": 20,
  "sort": [{ "price": "desc" }, { "_id": "desc" }],
  "query": { "match_all": {} },
  "search_after": [1999, "abc123"]
}

search_after 不依赖 offset，而是利用索引的有序性直接定位——从第 1 页跳转到第 1000 页的时间与第 1 页基本一致。

四、Spring Data Elasticsearch ​

RestHighLevelClient 已在 ES 7.x 后进入维护期，新项目推荐使用 ElasticsearchClient（新版 Java Client）或 Spring Data Elasticsearch：

xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-elasticsearch</artifactId>
</dependency>
java
// 文档实体
@Document(indexName = "products")
public class ProductDoc {
    @Id
    private Long id;

    @Field(type = FieldType.Text, analyzer = "ik_max_word")
    private String name;

    @Field(type = FieldType.Keyword)
    private String brand;

    @Field(type = FieldType.Integer)
    private Integer price;
}
java
// Repository 模式
public interface ProductRepository
        extends ElasticsearchRepository<ProductDoc, Long> {

    // 方法名自动生成查询
    List<ProductDoc> findByNameContainingAndPriceBetween(
        String keyword, Integer minPrice, Integer maxPrice);

    // 原生 DSL 查询
    @Query("{\"bool\": {\"must\": [{\"match\": {\"name\": \"?0\"}}]}}")
    List<ProductDoc> searchByName(String keyword);
}
java
// 批量写入
@Service
public class ProductIndexService {

    @Autowired
    private ElasticsearchOperations operations;

    public void bulkIndex(List<ProductDoc> products) {
        List<IndexQuery> queries = products.stream()
            .map(p -> new IndexQueryBuilder().withObject(p).build())
            .toList();
        operations.bulkIndex(queries, ProductDoc.class);
    }
}

五、生产环境运维考量 ​

5.1 分片设计 ​

索引创建时最重要的两个参数：

json
{
  "settings": {
    "number_of_shards": 3,       // 主分片数——创建后不可修改！
    "number_of_replicas": 1      // 每个主分片的副本数
  }
}
参数过小过大
number_of_shards单分片过大（>50GB），写入和查询都在一个分片上小分片过多，每个分片有固定开销（每个分片是一个 Lucene 实例）
number_of_replicas无冗余，节点故障导致数据丢失写入放大（每次写操作要同步到所有副本）

推荐值：单分片 ≤ 50GB，单节点分片总数 ≤ 20 × (堆内存 GB)。一个 100GB/天的日志索引，10 个主分片（每个 10GB/天）是合理的起点。

5.2 Refresh 与写入性能 ​

ES 的写入流程：写入内存 buffer → refresh（默认 1s）→ 可见可搜索。默认的 1s refresh 间隔意味着：

  • 写入后最多 1 秒即可被搜索到（近实时）
  • 每秒产生一个新的 Lucene 段文件，频繁的小段合并消耗 I/O

对于批量导入场景，关闭 refresh 可以大幅提升写入速度：

java
// 批量导入时的优化
BulkRequest bulkRequest = new BulkRequest();
bulkRequest.setRefreshPolicy(WriteRequest.RefreshPolicy.NONE);  // 不立即刷新
// ... 添加大量文档
bulkResponse = client.bulk(bulkRequest, RequestOptions.DEFAULT);

导入完成后手动触发一次 refresh 使数据可搜索。写入密集型场景下，将 refresh_interval 调整为 30s 或 -1（关闭自动 refresh），写入吞吐量可提升 3-5 倍。

5.3 堆内存配置 ​

ES 是 JVM 应用，但堆内存并非越大越好。Lucene 大量使用堆外内存（文件系统缓存），堆内存设太大会挤占 OS 的文件缓存空间：

bash
# jvm.options
-Xms4g
-Xmx4g                    # 堆内存不超过物理内存的 50%，上限 32GB
                          # >32GB 时 JVM 压缩指针失效，内存利用率反而下降

推荐：32GB 以内物理机 → 堆内存设为物理内存的 50%；更大物理机 → 堆内存上限 31GB（保持压缩指针有效）。

六、总结 ​

层次关键概念一句话
数据模型倒排索引、text vs keywordES 按词搜索，MySQL 按值查找——模型完全不同
查询 DSLbool/function_score/aggregationfilter 可缓存，must 贡献得分，function_score 叠加业务权重
分页search_after 替代 from+size深分页必须用 search_after，from+size 超过 10000 条就崩
Java 客户端Spring Data Elasticsearch新项目用 ElasticsearchClient 或 Spring Data，告别 RestHighLevelClient
运维分片规划、refresh 间隔、堆内存分片 ≤ 50GB，堆内存 ≤ 31GB，批量导入时关闭 refresh

ES 的强大与复杂并存。从理解倒排索引开始，掌握 DSL 的核心模式，然后逐渐深入到集群运维——这是后端开发者使用 ES 的有效学习路径。
