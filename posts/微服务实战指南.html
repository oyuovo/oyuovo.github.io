引言：你真的需要微服务吗？ ​

微服务架构在过去几年被广泛讨论，Spring Cloud Alibaba 生态也已相当成熟。但在投入微服务之前，有一个比「怎么实现」更重要的问题：你的项目真的需要微服务吗？

创业期 / MVP 验证  →  单体架构优先（快速迭代，低成本试错）
业务稳定增长      →  逐步拆分，先拆核心链路
资金充足 + 目标明确 →  可以直接选择微服务架构

微服务的代价是实实在在的：分布式事务的复杂性、网络延迟的不确定性、运维成本的指数级上升、调试排错的难度倍增。如果 2-3 人团队做一个内部管理系统，单体 + 模块化拆分往往是更务实的选择。

如果你确实需要微服务，本文覆盖 Spring Cloud Alibaba 技术栈的核心组件：Nacos → OpenFeign → Gateway → Sentinel → Seata，按实际项目中的使用顺序组织。

一、服务注册与发现：Nacos ​

1.1 注册中心解决了什么问题？ ​

在微服务架构中，服务实例的数量和地址是动态变化的（扩缩容、重启、故障转移）。硬编码 IP:Port 的调用方式无法应对这种动态性。注册中心的作用：

  1. 服务注册：每个服务启动时向注册中心报告自己的地址和元数据
  2. 服务发现：调用方从注册中心获取目标服务的可用实例列表
  3. 健康检查：通过心跳机制剔除不健康的实例
  4. 变更推送：服务实例发生变化时主动通知调用方

1.2 部署与集成 ​

使用 Docker 部署 Nacos（推荐使用 2.x 版本，gRPC 通信替代了旧版的 HTTP 心跳）：

bash
docker run -d \
  --name nacos \
  --env-file ./nacos/custom.env \
  -p 8848:8848 -p 9848:9848 -p 9849:9849 \
  --restart=always \
  nacos/nacos-server:v2.1.0-slim

端口说明

  • 8848：HTTP 控制台和 API 端口
  • 9848：gRPC 客户端通信端口（2.x 新增，服务发现依赖此端口）
  • 9849：gRPC 服务端通信端口

微服务侧集成两步完成：

xml
<!-- 每个需要注册的服务都要引入 -->
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-nacos-discovery</artifactId>
</dependency>
yaml
spring:
  application:
    name: order-service        # 服务名称，全局唯一标识
  cloud:
    nacos:
      server-addr: localhost:8848
      discovery:
        namespace: dev          # 隔离环境（dev/test/prod）
        group: DEFAULT_GROUP    # 服务分组

1.3 服务发现与负载均衡 ​

调用方通过服务名发起调用，底层由 LoadBalancer 选取具体实例：

java
@Service
public class OrderService {

    private final DiscoveryClient discoveryClient;

    public OrderService(DiscoveryClient discoveryClient) {
        this.discoveryClient = discoveryClient;
    }

    public UserDTO getUserById(Long userId) {
        // 1. 根据服务名获取实例列表（已自动过滤不健康实例）
        List<ServiceInstance> instances = discoveryClient.getInstances("user-service");

        // 2. 负载均衡选择（Spring Cloud LoadBalancer 基于 Round-Robin）
        ServiceInstance instance = instances.get(
            ThreadLocalRandom.current().nextInt(instances.size())
        );

        // 3. 发起 HTTP 调用
        URI uri = instance.getUri();
        return restTemplate.getForObject(uri + "/user/" + userId, UserDTO.class);
    }
}

实际项目中不会手动操作 DiscoveryClient ——下一节的 OpenFeign 封装了以上全部流程。

二、声明式 HTTP 调用：OpenFeign ​

OpenFeign 将远程调用从「手动拼接 URL + RestTemplate」简化为「本地接口方法调用」。它基于 Spring MVC 注解，学习成本极低。

2.1 基础使用 ​

java
// 1. 定义 Feign 客户端接口
@FeignClient(value = "user-service")   // value = 目标服务名
public interface UserClient {

    @GetMapping("/user/{id}")
    UserDTO getUserById(@PathVariable("id") Long id);

    @GetMapping("/users")
    List<UserDTO> queryByIds(@RequestParam("ids") List<Long> ids);

    @PostMapping("/user")
    UserDTO createUser(@RequestBody UserDTO user);
}

// 2. 像调用本地方法一样使用
@Service
public class OrderService {

    @Autowired
    private UserClient userClient;   // Spring 自动生成代理实现

    public OrderVO getOrderDetail(Long orderId) {
        Order order = orderMapper.selectById(orderId);
        UserDTO user = userClient.getUserById(order.getUserId());  // 透明远程调用
        return OrderVO.of(order, user);
    }
}
java
// 启动类上开启 Feign
@EnableFeignClients
@SpringBootApplication
public class OrderApplication {
    public static void main(String[] args) {
        SpringApplication.run(OrderApplication.class, args);
    }
}

2.2 连接池配置 ​

OpenFeign 默认使用 HttpURLConnection，不支持连接池，每次请求都建立新连接。生产环境必须替换：

xml
<!-- 使用 Apache HttpClient 5 作为底层 HTTP 客户端 -->
<dependency>
    <groupId>io.github.openfeign</groupId>
    <artifactId>feign-hc5</artifactId>
</dependency>
yaml
spring:
  cloud:
    openfeign:
      httpclient:
        hc5:
          enabled: true
      client:
        config:
          default:
            connect-timeout: 3000      # 连接超时
            read-timeout: 5000         # 读取超时（远程调用建议 ≥ 3s）

2.3 最佳实践：抽取公共 API 模块 ​

当服务 A 要调服务 B 的接口时，A 需要知道 B 的 DTO 和 FeignClient 定义。手动拷贝会导致定义不一致。标准做法是由服务提供方抽取独立的 API 模块：

order-service/
├── order-api/              ← 对外发布的接口模块
│   ├── client/
│   │   └── OrderClient.java     # FeignClient 定义
│   └── dto/
│       ├── OrderDTO.java
│       └── CreateOrderRequest.java
└── order-service/           ← 服务实现模块

调用方只需依赖 order-api，FeignClient 和 DTO 自动可用。如果 FeignClient 不在启动类的扫描包路径下，指定扫描路径：

java
@EnableFeignClients(basePackages = "com.example.order.api.client")

三、网关：Spring Cloud Gateway ​

网关是所有外部请求的入口，负责路由转发、身份校验、请求过滤。

3.1 为什么选 Spring Cloud Gateway？ ​

微服务网关的选择曾经是 Spring Cloud Gateway vs Netflix Zuul。Zuul 基于 Servlet 阻塞模型，已进入维护状态。Spring Cloud Gateway 基于 WebFlux 响应式编程，无需调优即可获得显著更优的吞吐量。新项目直接选 Gateway。

3.2 路由配置 ​

yaml
server:
  port: 8080
spring:
  application:
    name: gateway
  cloud:
    nacos:
      server-addr: localhost:8848
    gateway:
      routes:
        - id: order-route
          uri: lb://order-service     # lb = 负载均衡
          predicates:
            - Path=/api/order/**      # 路径匹配断言
          filters:
            - StripPrefix=1            # 转发前剥离 /api 前缀
        - id: user-route
          uri: lb://user-service
          predicates:
            - Path=/api/user/**
          filters:
            - StripPrefix=1

关键概念：

  • Predicate（断言）：决定请求是否匹配当前路由。Gateway 内置 12 种断言工厂（Path、Header、Query、Method、Cookie 等）
  • Filter（过滤器）：对请求/响应做处理。StripPrefix=1 表示转发前去掉一段路径，即 /api/order/create → /order/create
  • lb:// vs http://：lb:// 启用客户端负载均衡，自动从 Nacos 获取实例列表并轮询

3.3 全局登录校验 ​

网关是实施统一鉴权的理想位置。以下实现基于 JWT 的全局过滤器：

java
@Component
public class AuthGlobalFilter implements GlobalFilter, Ordered {

    @Autowired
    private JwtTool jwtTool;

    private final AntPathMatcher antPathMatcher = new AntPathMatcher();

    // 无需登录即可访问的路径
    private static final List<String> EXCLUDE_PATHS = List.of(
        "/api/auth/login",
        "/api/auth/register"
    );

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        ServerHttpRequest request = exchange.getRequest();
        String path = request.getURI().getPath();

        // 1. 判断是否需要登录拦截
        if (isExcluded(path)) {
            return chain.filter(exchange);
        }

        // 2. 获取并解析 Token
        String authHeader = request.getHeaders().getFirst("Authorization");
        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            return unauthorized(exchange, "未登录");
        }

        Long userId;
        try {
            userId = jwtTool.parseToken(authHeader.substring(7));
        } catch (Exception e) {
            return unauthorized(exchange, "Token 无效或已过期");
        }

        // 3. 将用户信息通过请求头传递给下游微服务
        ServerWebExchange mutatedExchange = exchange.mutate()
            .request(builder -> builder.header("X-User-Id", userId.toString()))
            .build();

        return chain.filter(mutatedExchange);
    }

    @Override
    public int getOrder() {
        return 0;  // 数字越小优先级越高
    }

    private boolean isExcluded(String path) {
        return EXCLUDE_PATHS.stream()
            .anyMatch(p -> antPathMatcher.match(p, path));
    }

    private Mono<Void> unauthorized(ServerWebExchange exchange, String message) {
        ServerHttpResponse response = exchange.getResponse();
        response.setStatusCode(HttpStatus.UNAUTHORIZED);
        response.getHeaders().setContentType(MediaType.APPLICATION_JSON);
        byte[] body = ("{\"code\":401,\"message\":\"" + message + "\"}").getBytes();
        return response.writeWith(Mono.just(response.bufferFactory().wrap(body)));
    }
}

下游微服务如何获取用户信息？

网关将 userId 写入请求头 X-User-Id，下游服务通过 Spring MVC 拦截器读取并存入 ThreadLocal：

java
String userId = request.getHeader("X-User-Id");
UserContext.setCurrentUser(Long.valueOf(userId));

注意：网关（WebFlux）中不要引入 Spring MVC 依赖，否则 DispatcherServlet 会接管路由。通过 @ConditionalOnClass(DispatcherServlet.class) 可以保证拦截器注册代码只在非网关服务中生效。

四、服务保护：Sentinel ​

4.1 雪崩问题的三个防线 ​

当调用链中某个服务出现故障或延迟时，如果没有保护机制，故障会沿调用链向上传播，最终导致整个集群不可用。Sentinel 提供三道防线：

机制作用类比
请求限流控制 QPS/并发数，超过阈值的请求直接拒绝景区限流：超过承载量不再放人
线程隔离为每个下游服务分配独立的线程池，慢服务不会耗尽公共线程船舱隔板：一个舱进水不影响其他舱
服务熔断统计错误率/慢调用比例，超过阈值时直接熔断该接口，快速失败断路器：短路保护

4.2 集成 Sentinel ​

xml
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-sentinel</artifactId>
</dependency>
yaml
spring:
  cloud:
    sentinel:
      transport:
        dashboard: localhost:8080       # Sentinel 控制台地址
      http-method-specify: true         # RESTful 接口用 Method+URL 作为资源名

启动后访问任意接口，即可在 Sentinel Dashboard 看到所有端点作为「簇点链路」资源。

4.3 限流与熔断配置 ​

在 Sentinel 控制台的「簇点链路」页面，对任意资源点击「流控」或「熔断」按钮即可配置规则。这些规则存储在内存中，服务重启后会丢失。生产环境必须持久化规则到 Nacos：

yaml
spring:
  cloud:
    sentinel:
      datasource:
        flow:                          # 限流规则
          nacos:
            server-addr: localhost:8848
            data-id: ${spring.application.name}-flow-rules
            group-id: SENTINEL_GROUP
            data-type: json
            rule-type: flow
        degrade:                       # 熔断规则
          nacos:
            server-addr: localhost:8848
            data-id: ${spring.application.name}-degrade-rules
            group-id: SENTINEL_GROUP
            data-type: json
            rule-type: degrade

4.4 Feign 调用失败降级 ​

限流/熔断只是拒绝请求，被拒绝的请求还需要一个降级处理——返回默认值或友好提示，而不是让用户看到 500 错误页。

java
@Slf4j
@Component
public class UserClientFallbackFactory implements FallbackFactory<UserClient> {

    @Override
    public UserClient create(Throwable cause) {
        log.error("UserClient 远程调用失败", cause);
        return new UserClient() {
            @Override
            public UserDTO getUserById(Long id) {
                // 返回兜底数据：如缓存的用户信息或空对象
                return UserDTO.fallback(id);
            }
        };
    }
}
java
@FeignClient(
    value = "user-service",
    fallbackFactory = UserClientFallbackFactory.class
)
public interface UserClient { ... }
yaml
# 启用 Sentinel 对 Feign 的监控
feign:
  sentinel:
    enabled: true

五、分布式事务：Seata ​

5.1 问题的本质 ​

当「创建订单」需要同时操作订单服务（写订单表）和库存服务（扣库存），两个操作必须全部成功或全部回滚。在分布式环境下，每个服务有自己独立的数据源和事务管理器，没有单机数据库的全局事务保证。

Seata 将整个过程抽象为：

  • 全局事务：整个业务流程
  • 分支事务：每个微服务内部的本地事务
  • TC（事务协调者）：独立的服务，维护全局和分支事务的状态，决定最终提交还是回滚

5.2 XA 模式 vs AT 模式 ​

Seata 提供两种主流事务模式：

维度XA 模式AT 模式
一致性强一致（遵循 XA 协议）最终一致
性能一阶段锁定数据库资源，事务期间不释放一阶段即提交，不长时间锁表
回滚方式数据库自身的事务回滚解析 undo_log 快照，反向 SQL
对数据库的要求数据库需支持 XA 协议不依赖 XA，适用于任意关系型数据库
入侵性零代码入侵需在每个业务的数据库中创建 undo_log 表

选择指南：

  • 对一致性要求极高（金融、账务）→ XA 模式
  • 追求性能和吞吐量，能接受短暂的数据不一致窗口 → AT 模式
  • 绝大多数互联网业务场景中，AT 模式是更务实的选择

5.3 AT 模式实战 ​

yaml
# 参与分布式事务的每个微服务
seata:
  registry:
    type: nacos
    nacos:
      server-addr: localhost:8848
      application: seata-tc-server    # TC 服务在 Nacos 中的服务名
  tx-service-group: myapp-tx-group
  service:
    vgroup-mapping:
      myapp-tx-group: default
  data-source-proxy-mode: AT           # 开启 AT 模式
sql
-- 每个参与 AT 模式的微服务数据库中都要创建此表
CREATE TABLE IF NOT EXISTS undo_log (
    id            BIGINT AUTO_INCREMENT PRIMARY KEY,
    branch_id     BIGINT NOT NULL,
    xid           VARCHAR(128) NOT NULL,
    context       VARCHAR(128) NOT NULL,
    rollback_info LONGBLOB NOT NULL,
    log_status    INT NOT NULL,
    log_created   DATETIME NOT NULL,
    log_modified  DATETIME NOT NULL,
    INDEX ux_undo_log (xid, branch_id)
);
java
// 发起全局事务的入口方法
@Service
public class OrderServiceImpl implements OrderService {

    @Autowired
    private OrderMapper orderMapper;
    @Autowired
    private StockClient stockClient;  // Feign 远程调用库存服务

    @Override
    @GlobalTransactional  // 关键：标注此方法为全局事务入口
    public Long createOrder(CreateOrderRequest request) {
        // 1. 本地：创建订单
        Order order = Order.from(request);
        orderMapper.insert(order);

        // 2. 远程：扣减库存
        stockClient.deduct(request.getProductId(), request.getQuantity());

        return order.getId();
        // 任何一个步骤失败，整个全局事务回滚
    }
}

AT 模式的工作流程：

  1. 一阶段：各服务执行本地 SQL → 记录 undo_log（数据快照）→ 立即提交本地事务
  2. 二阶段-提交：TC 通知各 RM 删除 undo_log（清理快照）
  3. 二阶段-回滚：TC 通知各 RM 根据 undo_log 执行反向 SQL 恢复数据

核心选型建议：

组件选型一句话理由
注册/配置中心Nacos服务发现 + 配置管理一体，Alibaba 生态首选
远程调用OpenFeign + HttpClient5声明式 + 连接池，性能与开发体验兼顾
网关Spring Cloud GatewayWebFlux 响应式，性能优于 Zuul
服务保护Sentinel限流/熔断/降级一体，控制台可视化
分布式事务Seata AT 模式最终一致 + 高性能，适合大多数业务场景

每引入一个组件都意味着额外的运维负担。在做技术选型时，「如无必要，勿增实体」是不变的准则。
