引言 ​

如果你使用 Spring Boot 超过半年，大概率遇到过以下场景：

  • 两个 Bean 互相依赖，容器能正常启动——但它是怎么打破「先有鸡还是先有蛋」的死锁的？
  • 在同一个类里调用带 @Transactional 的方法，事务不生效——为什么？
  • AOP 切面写好了，但 final 方法上的通知消失了——又为什么？

这些问题的答案都藏在 IoC 容器和 AOP 代理机制的底层设计中。本文假设你已经用过 @Autowired 和 @Transactional，聚焦于解释这些日常注解背后的「为什么」。

一、IoC 容器：不止于「别 new」 ​

1.1 控制反转的本质 ​

IoC（Inversion of Control）的核心思想可以用一句话概括：

对象的创建和管理权从程序代码转移到外部容器，调用方只需声明依赖，由容器负责装配。

对比两种方式：

java
// ❌ 传统方式：自己管理依赖
public class OrderService {
    private UserRepository userRepo = new UserRepositoryImpl();   // 硬编码 + 紧耦合
    private OrderRepository orderRepo = new OrderRepositoryImpl();
}

// ✅ IoC 方式：容器注入依赖
@Service
public class OrderService {
    @Autowired
    private UserRepository userRepo;      // 接口类型，具体实现由容器决定
    @Autowired
    private OrderRepository orderRepo;
}

IoC 不是一个技术实现，而是一种设计原则。DI（Dependency Injection，依赖注入）是 IoC 在 Spring 中的具体实现方式——容器通过构造器或 setter 将依赖「注入」到 Bean 中，而不是让 Bean 自己去查找或创建依赖。

1.2 Bean 生命周期：从定义到销毁 ​

理解 Bean 生命周期是排查 Spring 问题的必修课。一个 Bean 从创建到销毁经历以下关键节点：

1. 实例化（Instantiation）
      通过反射调用构造器创建对象实例
         │
         ▼
2. 属性赋值（Populate Properties）
      注入 @Autowired 标注的依赖
         │
         ▼
3. 初始化前（BeanPostProcessor.postProcessBeforeInitialization）
      @PostConstruct 注解的方法在此阶段之前执行
         │
         ▼
4. 初始化（Initialization）
      执行 InitializingBean.afterPropertiesSet() 和 init-method
         │
         ▼
5. 初始化后（BeanPostProcessor.postProcessAfterInitialization）
      ★ AOP 代理对象在此阶段生成 —— 返回的是代理对象而非原始 Bean
         │
         ▼
6. 就绪（Ready）
      Bean 可以被应用程序使用了
         │
         ▼
7. 销毁前（Destruction）
      @PreDestroy → DisposableBean.destroy() → destroy-method

关键理解：AOP 代理的生成发生在「初始化后」阶段。BeanPostProcessor.postProcessAfterInitialization 是 Spring 创建代理的入口——AbstractAutoProxyCreator 在这里判断 Bean 是否需要被代理，如果需要则创建 JDK 或 CGLIB 代理对象并返回，后续从容器中获取的 Bean 都是这个代理对象。

1.3 循环依赖：三级缓存如何破局 ​

考虑以下场景：

java
@Service
public class AService {
    @Autowired
    private BService bService;
}

@Service
public class BService {
    @Autowired
    private AService aService;
}

A 依赖 B，B 依赖 A。Spring 通过 三级缓存 解决这个问题：

缓存名称存储内容
一级缓存singletonObjects完全初始化好的 Bean
二级缓存earlySingletonObjects早期暴露的 Bean（已实例化但未完成属性填充）
三级缓存singletonFactoriesBean 的 ObjectFactory（可生成早期 Bean 引用的工厂）

解决流程（简化版）：

1. 容器创建 A
   → 实例化 A（构造器调用）
   → 将 A 的 ObjectFactory 放入三级缓存（提前暴露）
   → 开始填充 A 的属性 → 发现需要 B

2. 容器创建 B
   → 实例化 B（构造器调用）
   → 将 B 的 ObjectFactory 放入三级缓存
   → 开始填充 B 的属性 → 发现需要 A
   → 从三级缓存中找到 A 的 ObjectFactory，调用 getObject() 获得 A 的早期引用
   → 将 A 的早期引用放入二级缓存，同时从三级缓存中移除
   → B 完成属性填充和初始化

3. 回到 A
   → B 完全就绪，注入到 A
   → A 完成属性填充和初始化
   → A 从二级缓存移入一级缓存

为什么需要三级缓存？ 仅仅解决循环依赖只需要二级缓存。三级缓存的存在是为了处理 AOP 代理 的情况——如果 A 需要被代理（有切面配置），提前暴露的 ObjectFactory 可以先创建代理对象再返回，保证 B 注入的 A 和最终暴露的 A 是同一个代理实例。

构造器注入无法解决循环依赖

上述机制仅在 setter 注入（@Autowired 标注在字段或 setter 方法上）时有效。构造器注入在实例化阶段就需要所有依赖，此时 A 还没有被放入缓存，无法提前暴露。遇到构造器循环依赖时 Spring 会抛出 BeanCurrentlyInCreationException。这也是 Spring 团队推荐构造器注入的原因之一——它迫使你发现并消除循环依赖，而不是让框架替你掩盖设计问题。

二、AOP 代理：深入字节码层面 ​

2.1 JDK 动态代理 vs CGLIB 代理 ​

Spring AOP 有两种代理实现，选择逻辑如下：

java
// Spring 默认选择逻辑（AopAutoConfiguration）
if (target instanceof interface的实现) {
    // 使用 JDK 动态代理（基于接口）
    // 要求目标类必须实现至少一个接口
} else {
    // 使用 CGLIB（基于继承，生成目标类的子类）
}

从 Spring Boot 2.0 开始，默认使用 CGLIB 代理（配置项 spring.aop.proxy-target-class 默认为 true）。

两者的核心差异：

维度JDK 动态代理CGLIB 代理
代理方式基于接口，生成 $Proxy 类基于继承，生成目标类的子类
被代理要求必须实现接口不能是 final 类（无法继承）
方法要求代理接口中定义的方法不能被 final 修饰（无法覆写）
性能反射调用，早期被认为较慢生成字节码直接调用，启动时慢，运行时快
Spring 默认否（2.0+）是（2.0+）
为什么 Spring Boot 2.0 默认改为 CGLIB？

主要原因是 类型注入的便利性。如果使用 JDK 动态代理，从容器中获取 Bean 时只能用接口类型注入，不能用具体实现类注入——因为代理对象只实现了接口，不是实现类的子类。CGLIB 代理是目标类的子类，同时支持接口类型和实现类类型的注入，避免开发者在注入时遇到 NoSuchBeanDefinitionException。

2.2 自调用失效：最常见也最隐蔽的 AOP 陷阱 ​

这是 Spring AOP 中被问到最多的问题：

java
@Service
public class UserService {

    @Transactional
    public void doSomething() {
        // 数据库操作
    }

    public void outerMethod() {
        // 直接调用被 @Transactional 标注的方法
        this.doSomething();   // ❌ 事务不会生效！
    }
}

原因：this.doSomething() 调用的是 原始对象（raw object），而不是 Spring 容器中的 代理对象。AOP 是通过代理对象方法拦截实现的——调用代理对象的 doSomething() → 代理执行前置通知 → 调用目标方法 → 代理执行后置通知。直接 this.doSomething() 绕过了整个代理链。

outerMethod() 调用 this.doSomething():
  UserService (this) → doSomething()     ← 直接在原始对象上调用，无代理

如果是外部调用 userService.doSomething():
  调用者 → 代理对象 → 事务拦截器 → doSomething()  ← 经过代理链

三种解决方案：

java
// 方案一：通过注入自身获取代理对象（最常用）
@Service
public class UserService {
    @Autowired
    private UserService self;  // 注入的是代理对象

    public void outerMethod() {
        self.doSomething();    // ✅ 通过代理对象调用，事务生效
    }
}

// 方案二：抽到另一个 Service（推荐的架构改进）
@Service
public class OrderService {
    @Autowired
    private UserService userService;  // 天然就是代理对象

    public void outerMethod() {
        userService.doSomething();    // ✅ 不同类之间的调用，经过代理
    }
}

// 方案三：AopContext.currentProxy()（仅限特定场景）
@Service
public class UserService {
    public void outerMethod() {
        ((UserService) AopContext.currentProxy()).doSomething();  // ✅
    }
}
// 需要在 @EnableAspectJAutoProxy(exposeProxy = true) 中启用

最佳实践

方案二（抽到另一个 Service）是最干净的架构选择——它消除了自调用需求本身。方案一直接但给代码增加了一层间接性。选择时考虑：如果这是你在当前类中唯一需要自调用的方法，用方案一；如果自调用模式反复出现，说明业务逻辑应该拆分到新的 Service 中。

2.3 环绕通知实战：方法日志注解 ​

AOP 最常见的生产应用之一是 @MethodLog——记录方法的参数、返回值和执行时间：

java
// 自定义注解
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface MethodLog {
    String value() default "";        // 方法描述
    boolean logArgs() default true;   // 是否记录参数
    boolean logResult() default false; // 是否记录返回值
}
java
@Aspect
@Component
@Slf4j
public class MethodLogAspect {

    @Around("@annotation(methodLog)")
    public Object around(ProceedingJoinPoint joinPoint, MethodLog methodLog) throws Throwable {
        String methodName = joinPoint.getSignature().toShortString();
        Object[] args = joinPoint.getArgs();

        // 1. 前置：记录方法调用和参数
        if (methodLog.logArgs()) {
            log.info("[{}] 调用开始, 参数: {}", methodLog.value(), JSON.toJSONString(args));
        }

        long start = System.currentTimeMillis();
        try {
            // 2. 执行目标方法
            Object result = joinPoint.proceed();

            // 3. 返回通知：记录执行时间和返回值
            long elapsed = System.currentTimeMillis() - start;
            if (methodLog.logResult()) {
                log.info("[{}] 调用完成, 耗时: {}ms, 返回值: {}",
                    methodLog.value(), elapsed, JSON.toJSONString(result));
            } else {
                log.info("[{}] 调用完成, 耗时: {}ms", methodLog.value(), elapsed);
            }
            return result;
        } catch (Throwable e) {
            // 4. 异常通知：记录异常信息
            log.error("[{}] 调用异常, 耗时: {}ms", methodLog.value(),
                System.currentTimeMillis() - start, e);
            throw e;
        }
    }
}

使用效果——一行注解完成方法日志：

java
@MethodLog(value = "创建订单", logResult = true)
public OrderVO createOrder(CreateOrderRequest request) {
    // 业务逻辑...
    return orderVO;
}
// 日志输出：
// [创建订单] 调用开始, 参数: {"userId":1,"productId":100}
// [创建订单] 调用完成, 耗时: 342ms, 返回值: {"orderId":12345,"status":"CREATED"}

三、总结 ​

机制一句话总结排障关键词
IoC 容器对象的创建/管理/装配由容器负责，调用方只声明依赖NoSuchBeanDefinitionException → 检查组件扫描范围
三级缓存实例化→提前暴露→填充依赖→完成初始化，解决 setter 循环依赖构造器循环依赖 → 无法解决，重构设计
AOP 代理BeanPostProcessor 在初始化后阶段创建代理对象，拦截方法调用自调用失效 → this.method() 绕过了代理
JDK vs CGLIBSpring Boot 2.0+ 默认 CGLIB，基于继承生成子类代理final 方法无法增强 → 去掉 final 或改用接口代理

深入理解这些机制的意义在于：当 Spring 的行为不符合直觉时，你能沿着调用链溯源，而不是加一行 @Lazy 又一行 proxyTargetClass=true 碰运气。
