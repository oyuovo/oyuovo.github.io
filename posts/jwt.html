引言 ​

基于 Token 的无状态认证是现代 Web 应用的主流方案。JWT（JSON Web Token）是最广泛使用的 Token 实现，但「用上 JWT」和「安全地用对 JWT」之间有一道鸿沟——Token 存 localStorage 还是 httpOnly Cookie？刷新 Token 如何设计？签名的 none 算法攻击怎么防？这些问题在教程中经常被一笔带过，但在生产环境中每个都是安全隐患。

本文以 Spring Boot + Spring Security 为后端栈，实现一个完整的双 Token 认证方案，并覆盖最常见的 JWT 安全陷阱。

一、为什么是 Token，不是 Session？ ​

传统的 Session 认证中，服务器在内存中维护会话状态。这在单体应用中工作良好，但在分布式环境下带来两个问题：

问题Session 方案的应对Token 方案的应对
水平扩展时 Session 不共享Session 复制（开销大）或集中存储（Redis）不需要——Token 由客户端携带，服务端无状态
跨域请求中 Cookie 不共享CORS + withCredentials 复杂配置Token 通过 Authorization 头显式传递，不受同源策略限制

Token 认证的本质：将状态从服务端转移到客户端，以解析 Token 的 CPU 开销换取存储和扩展的便利。

二、JWT 的结构与签名原理 ​

JWT 由三部分组成，以 . 分隔：

eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJveXVvdm8ifQ.4K-xxxxxxxxxxxx
└───── Header ────┘ └─── Payload ───┘ └────── Signature ──────┘

2.1 Header ​

json
{
  "alg": "HS256",
  "typ": "JWT"
}

alg 声明签名算法。常见值：HS256（HMAC-SHA256，对称密钥）、RS256（RSA-SHA256，非对称密钥）。

2.2 Payload ​

json
{
  "sub": "12345",          // 主体（通常是用户 ID）
  "iat": 1701234567,       // 签发时间（Unix 时间戳）
  "exp": 1701238167,       // 过期时间
  "roles": ["USER"]        // 自定义字段（不要放敏感信息！）
}

Payload 不是加密的

Base64URL 编码 ≠ 加密。任何人都可以解码 JWT 的 Header 和 Payload。永远不要将密码、密钥、个人隐私信息放入 Payload。 如果你确实需要在 Token 中携带需要保密的数据，应该使用 JWE（JSON Web Encryption），而非普通的 JWT（更准确地说，应该是 JWS，JSON Web Signature）。

2.3 Signature ​

签名的作用是防篡改，不是加密：

signature = HMAC-SHA256(
    base64UrlEncode(header) + "." + base64UrlEncode(payload),
    secret_key
)

服务器收到 JWT 后，用相同的密钥重新计算签名。如果计算结果与 Token 中的签名不一致 → Token 被篡改 → 拒绝请求。

三、安全陷阱：none 算法攻击 ​

JWT 规范中允许 "alg": "none"，表示 Token 不签名。如果服务端的 JWT 库没有正确配置，攻击者可以将 Header 中的 alg 改为 none，并删除签名部分，服务端可能会直接信任这个未签名的 Token：

json
// 攻击者构造的恶意 Token
{"alg": "none", "typ": "JWT"}.{"sub": "admin", "roles": ["ADMIN"]}.

防范：

java
// jjwt 0.12+ 默认拒绝 none 算法
// 如果使用旧版本或手动解析，显式指定算法
Jwts.parser()
    .verifyWith(key)         // 必须指定密钥 → none 算法会直接报错
    .build()
    .parseSignedClaims(token);

永远不要接受 alg: none 的 Token。现代 JWT 库（jjwt 0.12+、auth0-java）已默认拒绝，但使用旧版本时需要注意。

四、Access Token + Refresh Token 双 Token 模式 ​

4.1 为什么需要两个 Token？ ​

如果只用一个 Token，你会面临一个困境：

  • Token 有效期长 → 泄漏后攻击窗口大，无法主动撤销
  • Token 有效期短 → 用户频繁重新登录，体验差

双 Token 模式解耦了这个问题：

Token 类型有效期存储位置用途
Access Token短（15-30 分钟）内存（JavaScript 变量）每次 API 请求携带
Refresh Token长（7-30 天）httpOnly Cookie仅用于获取新的 Access Token

Access Token 过期 → 用 Refresh Token 换新 → 用户无感知 → Access Token 泄漏的影响窗口只有 15 分钟。

4.2 Spring Boot 完整实现 ​

java
// JwtUtil.java — Token 生成与解析
@Component
public class JwtUtil {

    private final SecretKey accessTokenKey;
    private final SecretKey refreshTokenKey;

    // Access Token 和 Refresh Token 使用不同的密钥
    public JwtUtil(@Value("${jwt.access-secret}") String accessSecret,
                   @Value("${jwt.refresh-secret}") String refreshSecret) {
        this.accessTokenKey = Keys.hmacShaKeyFor(Decoders.BASE64.decode(accessSecret));
        this.refreshTokenKey = Keys.hmacShaKeyFor(Decoders.BASE64.decode(refreshSecret));
    }

    public String generateAccessToken(Long userId, List<String> roles) {
        return Jwts.builder()
                .subject(userId.toString())
                .claim("roles", roles)
                .issuedAt(new Date())
                .expiration(new Date(System.currentTimeMillis() + 15 * 60 * 1000)) // 15 分钟
                .signWith(accessTokenKey)
                .compact();
    }

    public String generateRefreshToken(Long userId) {
        return Jwts.builder()
                .subject(userId.toString())
                .issuedAt(new Date())
                .expiration(new Date(System.currentTimeMillis() + 7 * 24 * 3600_000L)) // 7 天
                .signWith(refreshTokenKey)
                .compact();
    }

    public Claims parseAccessToken(String token) {
        return Jwts.parser()
                .verifyWith(accessTokenKey)
                .build()
                .parseSignedClaims(token)
                .getPayload();
    }

    public Claims parseRefreshToken(String token) {
        return Jwts.parser()
                .verifyWith(refreshTokenKey)
                .build()
                .parseSignedClaims(token)
                .getPayload();
    }
}

为什么两个 Token 用不同密钥？

Access Token 频繁在 Authorization 头中传输，暴露面远大于 Refresh Token。如果共用密钥，一旦 Access Token 的签名密钥被泄漏（如通过日志），攻击者可以用它伪造理论上更安全的 Refresh Token。分离密钥将 Access Token 密钥泄漏的影响范围限制在短期 Token 内。

4.3 Spring Security 过滤器链 ​

java
@Component
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    @Autowired
    private JwtUtil jwtUtil;

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain filterChain)
            throws ServletException, IOException {

        String authHeader = request.getHeader("Authorization");
        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            filterChain.doFilter(request, response);
            return;
        }

        try {
            Claims claims = jwtUtil.parseAccessToken(authHeader.substring(7));
            Long userId = Long.valueOf(claims.getSubject());

            // 构建认证对象，设置到 SecurityContext
            UsernamePasswordAuthenticationToken authentication =
                new UsernamePasswordAuthenticationToken(userId, null, List.of());
            SecurityContextHolder.getContext().setAuthentication(authentication);
        } catch (JwtException e) {
            response.setStatus(401);
            response.setContentType("application/json;charset=UTF-8");
            response.getWriter().write("{\"code\":401,\"message\":\"Token 无效或已过期\"}");
            return;
        }

        filterChain.doFilter(request, response);
    }
}

五、Token 存储：localStorage vs httpOnly Cookie ​

这是一个经常被争论的问题：

存储方式XSS 风险CSRF 风险实现复杂度
localStorage❌ 高风险：任何 JS 都能读取✅ 天然免疫（不随请求自动发送）低
httpOnly Cookie✅ JS 无法读取❌ 随请求自动发送，需防 CSRF中

推荐方案：

  • Access Token → 存内存（JS 变量），不持久化。页面刷新后重新用 Refresh Token 获取，XSS 攻击窗口非常窄
  • Refresh Token → 存 httpOnly Cookie（SameSite=Strict + Secure），JS 完全无法访问，大幅降低 XSS 的损害范围
java
// 在登录接口中设置 Refresh Token Cookie
ResponseCookie refreshCookie = ResponseCookie.from("refresh_token", refreshToken)
    .httpOnly(true)          // JS 不可访问
    .secure(true)            // 仅 HTTPS 传输
    .sameSite("Strict")      // 仅同站请求携带（防 CSRF）
    .path("/api/auth")       // 仅在刷新接口路径下发送
    .maxAge(7 * 24 * 3600)   // 7 天
    .build();
response.addHeader(HttpHeaders.SET_COOKIE, refreshCookie.toString());

六、令牌撤销：当用户登出或被盗用时 ​

JWT 本质上是无状态的——服务器不会记录哪个 Token 是「有效的」。这带来了一个难题：如何主动撤销一个还在有效期内的 Token？

三种方案，复杂度递增：

方案原理适用场景
短有效期Access Token 15 分钟过期，Refresh Token 可服务端控制大多数应用，最简单
黑名单将需要撤销的 Token ID (jti) 存入 Redis，过滤器每次校验时查询需要在分钟级撤销 Token 的应用
版本号用户表中存一个 tokenVersion 字段，Token 中携带版本号，修改密码时版本号+1用户修改密码/退出所有设备时
java
// 黑名单方案：Token 撤销过滤器
@Component
public class TokenBlacklistFilter extends OncePerRequestFilter {

    @Autowired
    private StringRedisTemplate redisTemplate;

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain chain) {
        String jwtId = (String) request.getAttribute("jwt_jti");
        if (jwtId != null && Boolean.TRUE.equals(redisTemplate.hasKey("blacklist:" + jwtId))) {
            response.setStatus(401);
            return;
        }
        chain.doFilter(request, response);
    }
}

对于大多数应用，短有效期 Access Token + 服务端可控制的 Refresh Token 已经足够——Refresh Token 存储在数据库中，服务端可以随时删除它来阻止续期。

七、总结 ​

决策点推荐做法一句话理由
Token 有效期Access 15min + Refresh 7d平衡安全性与用户体验
签名算法HS256（单服务）或 RS256（微服务）对称/非对称按架构选择
Access Token 存储内存变量，不持久化最小化 XSS 攻击面
Refresh Token 存储httpOnly + Secure + SameSite CookieJS 不可访问
密钥管理Access 和 Refresh 使用不同密钥限制密钥泄漏的爆炸半径
none 算法永远拒绝现代 JWT 库默认安全
撤销策略短有效期 + Refresh Token 服务端可控复杂度与需求匹配

JWT 的安全性不是「用了就安全」，而是「正确地用了才安全」。双 Token 模式、正确的存储选择、密钥分离——这三项是 JWT 在生产环境中不出事的底线。
