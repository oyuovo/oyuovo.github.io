把「只会聊天」升级为「能真实调用工具」

第一部分：MCP 是什么？核心概念与工作流程 ​

MCP 的含义 ​

**MCP（Model Context Protocol，模型上下文协议）**是一套开放协议，用来把「外部能力」以统一方式接入到 AI Agent 中。
它解决的是一个非常具体的问题：让模型不仅能读你给的上下文，还能通过标准化接口去“调用工具 / 取资源 / 做动作”。

在 Cursor 里，你可以把 MCP 理解成：

  • 你配置并运行若干个 MCP Server
  • 每个 Server 暴露一组 Tools（工具） 和/或 Resources（资源）
  • Cursor 作为 MCP Client，把这些能力呈现给 Agent
  • Agent 在合适的时候发起工具调用（通常需要你确认），把结果再用于生成回答/改代码

MCP 的核心概念 ​

1）MCP Client ​

在我们的语境里，Cursor 就是 MCP Client。它负责：

  • 启动/连接 MCP Server
  • 拉取 server 的能力清单（tools/resources）
  • 在对话中代表 Agent 发起调用，并展示请求参数与返回结果

2）MCP Server ​

MCP Server 是一个进程（本地）或一个远程服务，它向 client 提供能力：

  • 本地进程常见：node / python / npx / docker run ...
  • 远程常见：HTTP/SSE 风格的 URL（不同实现可能叫 sse / streamableHttp 等）

3）Tools ​

Tool 是 “可被调用的动作”，通常带有：

  • 名称（例如 filesystem.readFile、github.search_issues）
  • 输入参数 schema（JSON 结构）
  • 输出（JSON/文本）

常见 Tool 类型：

  • 读写文件/目录（注意：很多 server 会做路径白名单限制）
  • 访问第三方 API（GitHub、数据库、CI、内部平台）
  • 浏览器自动化/抓取（例如 Playwright 类服务器）

4）Resources ​

Resource 更像 “可被读取的上下文素材”，例如：

  • 只读数据源（配置、文档、某个系统的查询结果）
  • 能被列出、按 URI 获取的内容

5）Transport ​

Cursor 常见支持两类：

  • stdio：启动本地命令，靠标准输入输出通信（最常见、最容易上手）
  • HTTP/SSE：连接远程服务（团队/企业内部工具常用）

你不需要记住协议细节，只要记住：stdio = 跑命令，HTTP/SSE = 连 URL。

MCP 的核心工作流程 ​

为了把“概念”落到 Cursor 的真实使用上，下面用 Cursor 官方文档的两张截图来对照关键步骤：
先出现「工具调用确认」，再出现「工具调用结果」。

图源：Cursor 官方文档 cursor.com/docs/mcp.md

把一次完整的链路拆开看：

  1. 你在 mcp.json 里配置 server（或通过 Cursor UI 配置）
  2. Cursor 启动/连接 server，并拉取 tools/resources 列表
  3. 你在 Composer 的 Agent 模式下提需求（例如“用 GitHub 工具查最近 10 条 issue”）
  4. Agent 判断需要用工具，发起 tool call（多数情况下会弹出权限确认）
  5. Cursor 执行调用，把结果回传给 Agent
  6. Agent 用返回结果生成最终输出（回答、改代码、产出报告等）

什么时候 MCP 真正“值回票价” ​

下面这些场景，MCP 的提升是质变：

  • 数据不在本地文件里：GitHub、数据库、线上日志、内部 API
  • 需要“可验证”的信息：能直接查询/列出/对比，而不是凭模型猜
  • 需要执行动作：创建 issue、跑脚本、批量改文件、生成报告

第二部分：Cursor 里的 MCP 配置与测试使用 ​

1）两种配置范围：项目级 vs 全局 ​

Cursor 支持把 MCP 配置放在两个位置（常用实践）：

  • 项目级：把配置放到项目根目录的 .cursor/mcp.json
    适合：团队共享、项目专用工具（建议 commit，但不要 commit 明文密钥）
  • 全局级：把配置放到用户目录的 ~/.cursor/mcp.json
    适合：个人通用工具（例如个人 GitHub token）

在 Windows 上，全局路径通常是：

  • %USERPROFILE%\.cursor\mcp.json

2）用 .cursor/mcp.json 配一个最小可用示例（stdio 本地命令） ​

在项目根目录创建文件：.cursor/mcp.json

示例（文件系统 server，限制在当前项目目录）：

json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "${workspaceFolder}"
      ]
    }
  }
}

要点：

  • mcpServers 是根字段，少了它 Cursor 可能直接忽略整份配置
  • args 用数组而不是一整条字符串，更稳定
  • 路径尽量用 ${workspaceFolder} 这类插值，避免手写硬编码

3）关于密钥的配置 ​

以 GitHub server 为例（不要把 token 写死在文件里）：

json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "${env:GITHUB_TOKEN}"
      }
    }
  }
}

然后在你的系统环境变量里配置 GITHUB_TOKEN（或用 PowerShell 当前会话临时设置）：

powershell
$env:GITHUB_TOKEN="ghp_xxx"

习惯建议：把 token 做成最小权限、可随时吊销的专用 token；并把 .cursor/mcp.json 当成“基础设施配置”来审查。

4）改完配置后要做什么：重启与检查工具列表 ​

MCP server 通常在 Cursor 启动时加载。因此改了 mcp.json 后，建议：

  • 完全退出并重启 Cursor
  • 打开设置页里 MCP 相关区域，确认 server 状态为已连接，并能看到 tool 列表

如果 server 支持认证流程（OAuth 或内置 auth 工具），按提示完成即可。

5）在 Cursor 里测试 MCP：一段“可复制的验收提示词” ​

在 Composer 中切换到 Agent（不是 Ask），然后直接发这段：

text
列出你当前可用的 MCP tools（按 server 分组）。

然后使用 filesystem 的工具：
1) 列出项目根目录下的顶层文件/文件夹
2) 读取 docs/posts 目录下任意一篇文章的前 30 行
最后告诉我：哪些工具调用需要人工确认、哪些可以设置为自动运行。

看到以下信号基本就算成功：

  • 能列出 MCP tools
  • 能实际读到文件内容
  • 对话里能展开看到 tool call 的参数与返回值

6）常见问题排查 ​

“Server 没出现 / Tools 列表为空” ​

  • 检查 .cursor/mcp.json 是否是合法 JSON（尤其逗号、引号）
  • 检查根字段是否为 mcpServers
  • 重启 Cursor（不要只关窗口）

“command not found / npx 找不到” ​

  • 本机需要可用的 Node.js（很多社区 MCP server 依赖 Node 18+）
  • 尝试在终端运行同样命令，确认环境变量 PATH 没问题

“Tools 出现了，但 Agent 不会用” ​

  • 确保你在 Composer 的 Agent 模式下
  • 在提示词里显式点名工具/任务（例如“使用 github 工具搜索 issue”）

“认证失败 / 401 / 403” ​

  • token 权限不够或过期
  • 变量没生效：确认 ${env:...} 对应的环境变量确实存在

7）安全建议 ​

  • 不要 commit 明文密钥（token、API key、cookie）
  • 尽量用 ${env:XXX} 引用环境变量
  • 把 .cursor/mcp.json 的变更当成“脚本/CI 配置”级别来 code review
  • 只安装/启用你信任的 server（MCP server 本质上是在你的环境里跑命令/访问网络）

小结 ​

  • MCP 的价值在于：把外部能力标准化接入 Agent，让“写代码”变成“能查、能跑、能验证”的闭环
  • 在 Cursor 里落地，核心就两件事：写好 mcp.json + 在 Composer/Agent 里验收工具调用
