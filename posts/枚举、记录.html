引言 ​

枚举和 Record 是 Java 中两个被严重低估的语法特性。很多开发者只用枚举来定义常量集、用 Record 来替代 @Data 注解——这远远低估了它们的能力。枚举可以有方法、实现接口、甚至有抽象方法；Record 不仅是不可变 DTO，还能在 Spring Boot 中替代 @ConfigurationProperties 的类定义。本文聚焦于这些不显而易见但极具实用价值的用法。

一、枚举：不只是常量列表 ​

1.1 枚举是完整的类 ​

Java 的 enum 本质上是一个继承自 java.lang.Enum 的 final 类，每个枚举常量是该类的一个 public static final 实例。这意味着它可以有构造器、字段、方法和接口实现：

java
public enum PaymentMethod {
    CREDIT_CARD("信用卡", true),
    DEBIT_CARD("借记卡", true),
    BANK_TRANSFER("银行转账", false),
    CASH_ON_DELIVERY("货到付款", false);

    private final String displayName;
    private final boolean supportsRefund;   // 是否支持退款

    PaymentMethod(String displayName, boolean supportsRefund) {
        this.displayName = displayName;
        this.supportsRefund = supportsRefund;
    }

    public String getDisplayName() { return displayName; }
    public boolean supportsRefund() { return supportsRefund; }
}

永远不要依赖 ordinal()：ordinal() 返回枚举常量在定义中的位置（从 0 开始）。一旦调整枚举常量的顺序，所有依赖 ordinal() 的代码静默出 Bug——编译器不会发出任何警告。用显式字段替代 ordinal()。

1.2 枚举策略模式：消除 switch ​

这是枚举最强大但不为人知的用法——用抽象方法替代 switch-on-enum：

java
// ❌ switch-on-enum：每次新增操作都要修改所有 switch
public enum Operation {
    ADD, SUBTRACT, MULTIPLY, DIVIDE;
}
// ... 在 10 个不同的地方写 switch(operation) { case ADD: ... }

// ✅ 枚举策略模式：操作逻辑封装在枚举内部
public enum Operation {
    ADD {
        @Override
        public double apply(double x, double y) { return x + y; }
    },
    SUBTRACT {
        @Override
        public double apply(double x, double y) { return x - y; }
    },
    MULTIPLY {
        @Override
        public double apply(double x, double y) { return x * y; }
    },
    DIVIDE {
        @Override
        public double apply(double x, double y) {
            if (y == 0) throw new ArithmeticException("除数不能为零");
            return x / y;
        }
    };

    public abstract double apply(double x, double y);
}

// 使用
double result = Operation.ADD.apply(3, 5);  // 8

新增操作 = 新增一个枚举常量 + 实现抽象方法。编译器会强制你为每个新常量实现 apply()，不存在「忘了加 case」的运行时 Bug。这是开闭原则在枚举层面的自然实现。

1.3 EnumMap 和 EnumSet：专为枚举优化的集合 ​

java
// ❌ HashMap<Day, String> —— 底层是数组+链表/红黑树，性能一般
Map<Day, String> schedule = new HashMap<>();

// ✅ EnumMap<Day, String> —— 底层是纯数组，操作复杂度严格 O(1)
Map<Day, String> schedule = new EnumMap<>(Day.class);

EnumMap 在内部使用数组而非哈希表，数组索引 = 枚举的 ordinal() 值。不需要计算 hash、不需要处理碰撞——理论上是最快的 Map 实现。EnumSet 同理，用位向量（bitset）替代数组，一个 long 就能表示最多 64 个枚举常量的集合：

java
// EnumSet 内部是一个 64 位的 long，每个 bit 对应一个枚举常量
EnumSet<Day> weekend = EnumSet.of(Day.SATURDAY, Day.SUNDAY);
// 内存占用: 8 bytes (一个 long) vs HashSet 的 ~120 bytes

法则：当 Map 的 Key 或 Set 的元素是枚举类型时，无脑选择 EnumMap / EnumSet。

二、Record：不可变数据载体 ​

2.1 Record 的本质 ​

java
// 一行代码
public record Point(int x, int y) {}

// 等同编译器自动生成：
// - private final int x; private final int y;
// - 全参构造器（canonical constructor）
// - x() 和 y() 访问器方法（不是 getX()！）
// - equals()、hashCode()、toString()
// - 类本身被 final 修饰，不可被继承

Record 的三个设计约束（不可绕过）：

  1. 所有字段都是 private final——不可变
  2. 不能继承其他类——Record 已经继承了 java.lang.Record
  3. 不能被继承——Record 类本身是 final 的

这些约束使 Record 成为 DTO（数据传输对象）的理想载体。

2.2 紧凑构造器：参数校验 ​

java
public record CreateOrderRequest(
    Long productId,
    Integer quantity
) {
    // 紧凑构造器——在字段赋值之前执行
    public CreateOrderRequest {
        if (productId == null || productId <= 0) {
            throw new IllegalArgumentException("productId 必须为正数");
        }
        if (quantity == null || quantity < 1) {
            throw new IllegalArgumentException("quantity 至少为 1");
        }
    }
}

紧凑构造器的参数列表是隐式的——它处理的是所有组件参数。在紧凑构造器执行完后，编译器自动生成标准赋值代码。

2.3 Spring Boot 集成：@ConfigurationProperties ​

Record 的不可变特性与 @ConfigurationProperties 天然契合：

java
@ConfigurationProperties(prefix = "app.storage")
@ConstructorBinding    // 通过构造器注入而非 setter
public record StorageProperties(
    String endpoint,
    String bucket,
    String accessKey,
    String secretKey,
    @DefaultValue("10") int maxUploadSizeMb
) {}
properties
app.storage.endpoint=https://oss.example.com
app.storage.bucket=myapp-prod
app.storage.access-key=AK123456
app.storage.secret-key=${STORAGE_SECRET}

配置对象在创建后不可变——线程安全、可以被缓存、可以在多个组件之间共享而不担心被意外修改。这正是 Record 比 @Data + Lombok 更适合配置类的原因。

2.4 Record 的序列化 ​

Record 的序列化与其他类不同：反序列化时，只有当 JSON 中的字段与 Record 的组件完全匹配时才能成功——没有无参构造器，没有 setter。

java
// Jackson 2.12+ 原生支持 Record 序列化和反序列化
ObjectMapper mapper = new ObjectMapper();
String json = mapper.writeValueAsString(new Point(3, 5));
Point p = mapper.readValue(json, Point.class);

如果 JSON 中有 Record 没有定义的字段 → 反序列化失败（默认行为）。这比传统的 POJO 更安全——你不会因为拼写错误而静默丢失数据。

三、总结 ​

特性推荐用法一句话
枚举字段显式字段替代 ordinal()顺序变了编译器不会救你
枚举策略抽象方法替代 switch新增常量时编译器强制实现
EnumMap/EnumSet枚举做 Key/元素时无条件使用纯数组实现，O(1) 无碰撞
Record 用途DTO、配置类、API 返回体不可变 + equals/hashCode 自动生成
Record + Spring@ConstructorBinding配置类注入的最佳实现方式

枚举和 Record 都体现了 Java 从繁琐走向简洁的趋势。它们不是「偶尔用一下」的语法糖，而是在正确的场景下能显著提升代码质量的核心工具。
