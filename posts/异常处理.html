引言 ​

异常处理是 Java 中看似简单、实则极易用错的语言特性。吞掉异常、在循环中使用 try-catch、将 checked exception 无脑包装为 RuntimeException——这些模式在每个 Java 项目的代码审查中都会反复出现。本文从异常的底层分类出发，逐步延伸到 Spring Boot 中的全局异常处理架构。

一、异常体系的根基 ​

1.1 Throwable 的三层继承树 ​

Throwable
├── Error — 系统级错误，程序无力处理
│   ├── OutOfMemoryError
│   ├── StackOverflowError
│   └── NoClassDefFoundError
│
└── Exception — 程序可以捕获和处理
    ├── RuntimeException（未检查异常 / unchecked）
    │   ├── NullPointerException
    │   ├── IllegalArgumentException
    │   └── IndexOutOfBoundsException
    │
    └── 非 RuntimeException（已检查异常 / checked）
        ├── IOException
        ├── SQLException
        └── ReflectiveOperationException

核心区分：

  • Error：不要试图捕获。OutOfMemoryError 被捕获后程序状态不确定，继续运行只会产生更多错误
  • RuntimeException：程序逻辑错误的表现——应该通过代码修复（如空值检查），而非 try-catch 掩盖
  • Checked Exception：编译器强制调用方处理——强迫开发者思考异常情况

1.2 Checked vs Unchecked：永恒的设计争论 ​

java
// Checked Exception：调用方必须处理
public String readFile(String path) throws IOException {
    return Files.readString(Path.of(path));
}

// Unchecked Exception：调用方可选处理
public void validate(User user) {
    if (user.getName() == null) {
        throw new IllegalArgumentException("用户名不能为空");
    }
}

选择法则：

场景使用原因
调用方有可行的恢复方案Checked强迫调用方处理，如重试、回退到默认值
调用方无法恢复（程序 Bug）Unchecked如空指针、非法参数，应该直接失败并通知开发者
资源操作（I/O、数据库）Checked网络中断、磁盘满等情况调用方需要知道

Spring 的设计哲学：全部转为 RuntimeException。Spring 认为大多数异常在业务层无法恢复，应该向上传播到框架统一处理。JdbcTemplate 将 SQLException 转换为 DataAccessException（RuntimeException 子类）就是这个理念的体现。

二、try-with-resources：告别 finally 中的资源关闭 ​

Java 7 引入的 try-with-resources 是异常处理领域最重要的语法改进：

java
// ❌ Java 7 之前：资源关闭代码量远超业务代码
BufferedReader reader = null;
try {
    reader = new BufferedReader(new FileReader("data.txt"));
    String line;
    while ((line = reader.readLine()) != null) {
        process(line);
    }
} catch (IOException e) {
    log.error("读取文件失败", e);
} finally {
    if (reader != null) {
        try {
            reader.close();    // close() 也可能抛异常！
        } catch (IOException e) {
            log.error("关闭文件失败", e);
        }
    }
}

// ✅ try-with-resources：资源自动关闭 + 异常抑制链
try (BufferedReader reader = new BufferedReader(new FileReader("data.txt"))) {
    String line;
    while ((line = reader.readLine()) != null) {
        process(line);
    }
} catch (IOException e) {
    log.error("读取文件失败", e);
}
异常抑制（Suppressed Exceptions）

如果 try 块体抛出了异常 A，close() 也抛出了异常 B，老式写法中异常 B 会覆盖异常 A（finally 块通常不捕获 close 的异常）。try-with-resources 中，异常 A 是主异常，异常 B 被作为 suppressed exception 附加到 A 上，通过 ex.getSuppressed() 可以获取。这意味着你不会因为资源关闭失败而丢失真正的业务异常。

三、异常设计最佳实践 ​

3.1 不要吞掉异常 ​

java
// ❌ 最恶劣的模式：空 catch 块
try {
    doSomething();
} catch (Exception e) {
    // 什么都没发生.jpg
}

// ❌ 只打印不传播：调用方不知道出错了
try {
    doSomething();
} catch (Exception e) {
    e.printStackTrace();    // 去哪了？容器日志里？控制台？
}

// ✅ 最少要记录日志
try {
    doSomething();
} catch (Exception e) {
    log.error("doSomething 执行失败, userId={}", userId, e);  // 记录上下文
    throw new ServiceException("操作失败", e);                 // 向上传播
}

3.2 自定义异常层次 ​

java
// 所有业务异常的基类
public class BusinessException extends RuntimeException {
    private final int code;
    public BusinessException(int code, String message) {
        super(message);
        this.code = code;
    }
    public int getCode() { return code; }
}

// 具体业务异常
public class UserNotFoundException extends BusinessException {
    public UserNotFoundException(Long userId) {
        super(10001, "用户不存在: " + userId);
    }
}

public class InsufficientBalanceException extends BusinessException {
    public InsufficientBalanceException(BigDecimal required, BigDecimal available) {
        super(20001, String.format("余额不足，需要: %s, 可用: %s", required, available));
    }
}

自定义异常层次的价值：调用方可以精确捕获感兴趣的错误类型，框架层可以映射错误码为 HTTP 状态码。

四、Spring Boot 全局异常处理 ​

前后端分离项目中，Controller 层不应该散布 try-catch：

java
@RestControllerAdvice
@Slf4j
public class GlobalExceptionHandler {

    // 业务异常 → 返回业务错误码
    @ExceptionHandler(BusinessException.class)
    public ApiResponse<?> handleBusiness(BusinessException ex) {
        log.warn("业务异常: code={}, message={}", ex.getCode(), ex.getMessage());
        return ApiResponse.fail(ex.getCode(), ex.getMessage());
    }

    // 参数校验失败（@Valid 触发）
    @ExceptionHandler(MethodArgumentNotValidException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public ApiResponse<?> handleValidation(MethodArgumentNotValidException ex) {
        String detail = ex.getBindingResult().getFieldErrors().stream()
            .map(e -> e.getField() + ": " + e.getDefaultMessage())
            .collect(Collectors.joining("; "));
        return ApiResponse.fail(400, detail);
    }

    // 兜底：未预料的异常
    @ExceptionHandler(Exception.class)
    @ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)
    public ApiResponse<?> handleUnknown(Exception ex) {
        log.error("未处理异常", ex);
        return ApiResponse.fail(500, "服务器内部错误");
    }
}

异常处理方法的匹配顺序：从具体到宽泛——先匹配 BusinessException，再 MethodArgumentNotValidException，最后 Exception 作为兜底。这与 try-catch 的多 catch 块逻辑一致。

五、总结 ​

实践推荐做法一句话
资源管理try-with-resourcesJava 7+ 无脑选，自动关闭 + 异常抑制
异常类型选择可恢复用 checked，不可恢复用 uncheckedSpring 生态倾向于全部 unchecked
空 catch绝不允许至少记录日志
异常传播记录上下文 + 包装 + 抛出原始异常信息不能丢（cause chain）
全局处理@RestControllerAdvice集中的错误码映射，Controller 保持清爽

异常处理的质量直接影响线上问题的排查效率。一个被吞掉的异常可能需要数小时来定位——而良好的异常传播和日志记录能在几分钟内锁定问题。
