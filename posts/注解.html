引言 ​

注解是 Java 中最「安静」的语言特性——加一个 @Override 或 @Autowired，背后的行为差异巨大，但语法本身没有任何变化。理解注解需要区分两个层次：如何定义注解（元注解、参数、默认值）和如何处理注解（运行时反射 vs 编译期 APT）。大多数开发者只接触了前者——后者是 Lombok、MapStruct、Dagger 等工具的基石。

一、注解的定义与元注解 ​

1.1 一个完整的自定义注解 ​

java
@Target({ElementType.METHOD, ElementType.TYPE})   // 可用于方法和类
@Retention(RetentionPolicy.RUNTIME)                // 保留到运行时
@Documented                                        // 包含在 Javadoc 中
public @interface MethodLog {
    String value() default "";                     // 最常用参数，命名为 value 可省略参数名
    boolean logArgs() default true;
    long timeout() default -1;                     // -1 表示不超时
}

1.2 四个元注解的作用 ​

元注解作用常见取值注意事项
@Target限定注解的使用范围TYPE, METHOD, FIELD, PARAMETER, CONSTRUCTOR可组合多个，必须设
@Retention控制注解的存活阶段SOURCE（编译期丢弃）, CLASS（保留到 class 文件）, RUNTIME（运行时可用）自定义注解几乎总是 RUNTIME
@Repeatable允许同一位置重复使用注解类 + 容器注解类名仅在确实需要重复使用时才添加
@Inherited子类是否继承父类注解—仅对 TYPE 有效，仅对 class 继承有效（interface 无效）

1.3 注解参数的类型限制 ​

注解参数的类型被严格限制为：基本类型、String、Class、枚举、注解，以及以上类型的数组。不允许包装类型（Integer 不可以，int 可以）。所有值必须在编译时确定——这是注解区别于配置文件的关键。

二、运行时注解处理：Spring 的反射模式 ​

Spring 在容器启动时通过反射扫描注解，决定 Bean 的创建和装配方式。我们可以用同样的模式自定义校验框架：

java
// 1. 定义校验注解
@Target(ElementType.FIELD)
@Retention(RetentionPolicy.RUNTIME)
public @interface ValidateRange {
    int min() default 0;
    int max() default Integer.MAX_VALUE;
    String message() default "值超出范围";
}

// 2. 注解处理器
public class AnnotationValidator {

    public static <T> List<String> validate(T object) {
        List<String> errors = new ArrayList<>();
        Class<?> clazz = object.getClass();

        for (Field field : clazz.getDeclaredFields()) {
            ValidateRange annotation = field.getAnnotation(ValidateRange.class);
            if (annotation == null) continue;

            field.setAccessible(true);
            try {
                int value = field.getInt(object);   // 反射获取值
                if (value < annotation.min() || value > annotation.max()) {
                    errors.add(field.getName() + ": " + annotation.message());
                }
            } catch (IllegalAccessException e) {
                // 记录错误
            }
        }
        return errors;
    }
}

// 3. 使用
public class CreateOrderRequest {
    @ValidateRange(min = 1, max = 100, message = "数量必须在 1-100 之间")
    private int quantity;
}

List<String> errors = AnnotationValidator.validate(request);
// → ["quantity: 数量必须在 1-100 之间"]

这正是 JSR 303 Bean Validation（@NotNull、@Min、@Max）的工作方式——注解 + 反射处理器。Hibernate Validator 是这套模式的工业级实现。

三、编译期注解处理：APT 与 AbstractProcessor ​

运行时注解的局限性：反射在 JVM 启动后才执行，错误只能在运行时暴露，且有性能开销。编译期注解处理（APT, Annotation Processing Tool）在编译期间扫描注解并生成新的源代码文件，零运行时开销。

3.1 Lombok 是如何工作的？ ​

Lombok 的 @Data、@Getter、@Builder 都是 SOURCE 级别的注解（@Retention(RetentionPolicy.SOURCE)）：

Java 源码 → 编译器调用 Lombok 的 AbstractProcessor → 
生成 getter/setter/constructor 的 AST 节点 → 合并到 class 文件 → 注解被丢弃

编译完成后，@Data 注解不在 class 文件中，运行时没有任何 Lombok 的痕迹。

3.2 自定义编译期处理器 ​

java
@SupportedAnnotationTypes("com.example.annotation.Builder")
@SupportedSourceVersion(SourceVersion.RELEASE_17)
@AutoService(Processor.class)  // Google AutoService 自动注册
public class BuilderProcessor extends AbstractProcessor {

    @Override
    public boolean process(Set<? extends TypeElement> annotations,
                           RoundEnvironment roundEnv) {
        for (Element element : roundEnv.getElementsAnnotatedWith(Builder.class)) {
            TypeElement typeElement = (TypeElement) element;
            String className = typeElement.getSimpleName() + "Builder";

            // 用 JavaPoet 生成 Builder 类源码
            TypeSpec builderClass = TypeSpec.classBuilder(className)
                .addModifiers(Modifier.PUBLIC, Modifier.STATIC)
                // ... 为每个字段生成 setter 方法、build() 方法
                .build();

            JavaFile javaFile = JavaFile.builder(
                processingEnv.getElementUtils().getPackageOf(typeElement).toString(),
                builderClass
            ).build();

            javaFile.writeTo(processingEnv.getFiler());  // 输出到 generated-sources
        }
        return true;
    }
}

编译后，target/generated-sources/annotations/ 下会生成新的 Java 文件。IDE 和编译工具链自动识别这个目录，生成的代码与手写代码一样参与编译和类型检查。

3.3 运行时 vs 编译期：如何选择？ ​

维度运行时注解（Spring 模式）编译期注解（Lombok/APT 模式）
处理时机JVM 启动后，反射编译期间，生成新代码
性能开销启动时扫描 + 反射零运行时开销
错误发现时机运行时（启动/测试时）编译时（IDE 实时反馈）
实现复杂度中等高（需了解 AST 或 JavaPoet）
典型应用DI 注入、AOP、ValidationLombok、MapStruct、Dagger、AutoService

如果你只需要在运行期读取注解值做决策 → 运行时注解。如果你需要在编译期基于注解生成重复性代码 → APT。

四、总结 ​

概念一句话
元注解@Target 限制范围，@Retention 控制生命周期——自定义注解必须设这两个
运行时处理反射读取注解 → 执行逻辑；Spring 的 DI 和 AOP 都基于这个模式
APT编译期间扫描注解 → 生成新源码；Lombok、MapStruct 都基于这个机制
注解 vs 配置注解更内聚（紧贴代码），配置文件更灵活（无需重新编译）

注解本身不包含任何逻辑——它只是一个标签。注解的力量来自处理注解的代码。理解运行时反射处理和编译期 APT 处理两种模式，是使用和设计注解框架的关键。
